# PLANS.md

이 문서는 MVP1에서 MVP2 이후로 이어지는 전체 로드맵과, 각 단계의 진입 조건을 정의한다. 지금 당장의 작업 순서는 `exec-plans/`가 담당하고, 이 문서는 그보다 상위의 "언제 다음 단계로 넘어가는가"를 다룬다.

---

## 1. 단계 개요

```
MVP1 (현재)  →  MVP1 안정화  →  MVP2  →  이후
```

| 단계 | 핵심 목표 | 상태 |
|---|---|---|
| MVP1 | 전체 공통 레아이웃(좌측 사이드바, 최상단 헤더), 로그인, 대시보드(레이아웃만, 실기능은 추후 예정), 일정, 회의실, 프로필/조직도 실기능 확보 | 진행 전 (설계 완료, 개인 실험 단계) |
| MVP1 안정화 | 3인 실험 결과를 바탕으로 담당자 배정 후 실제 서비스 가능한 수준으로 통합 | 착수 전 |
| MVP2 | AI 챗봇/오케스트레이션, 관리자 CRUD, 알림 | 설계 전 |

## 2. MVP1 범위 (확정, `product-specs/index.md` 기준)

- **실기능**: 로그인/로그아웃, 캘린더(일정 CRUD), 회의실(조회/예약/취소), 마이페이지(이메일/전화번호/활성상태 수정), 조직도(조회), 대시보드(AI 요약 제외 위젯), 테마 선택
- **디자인만**: 대시보드 AI 요약, 관리 탭(직원/팀 관리)
- **제외(시딩으로 대체)**: 관리자 직원/팀 CRUD API
- **완전 이연**: AI 비서 전체(FR-029~035)

## 3. MVP1 → MVP1 안정화 진입 조건

지금은 3인이 각자 전체 구조를 독립적으로 구현해보는 실험 단계다(`AGENTS.md` 8번). 다음 조건이 충족되면 "안정화" 단계로 넘어간다:

- [ ] 3인 모두 자신의 구현 버전에서 4개 도메인(`auth`/`user`/`schedule`/`meetingroom`) 핵심 플로우가 동작함을 확인
- [ ] `exec-plans/tech-debt-tracker.md`의 "상" 심각도 항목(access/refresh token 만료시간 등) 해소
- [ ] 도메인별 담당자 배정 (`AGENTS.md` 8번 갱신)
- [ ] 3인의 구현 방식 차이를 비교해 하나의 기준 구현으로 통합할 방법 결정 (이 프로젝트 특유의 단계 — 아래 4번 참고)

## 4. "개인 실험 → 통합" 방식 (이 프로젝트 특유의 진행 방식)

3인이 각자 독립적으로 동일한 설계 문서를 기반으로 구현하는 것은 이 프로젝트의 의도된 진행 방식이다(Harness Engineering 검증 목적도 겸함, `프로젝트 소개` 문서의 "AI 협업 개발 프로세스 구축" 참고). 안정화 단계에서는:

1. 3개의 독립 구현을 비교해, 설계 문서(`ARCHITECTURE.md`, `product-specs/`)를 얼마나 일관되게 따랐는지 확인한다. 구현 차이가 크게 난 지점은 설계 문서 자체가 모호했다는 신호이므로, 문서를 먼저 보완한다.
2. 최종적으로 하나의 코드베이스로 합칠지, 아니면 가장 안정적인 한 버전을 기준으로 나머지를 리뷰/보완하는 방식으로 갈지는 이 단계에서 팀이 결정한다(현재 미정).

## 5. MVP2 진입 조건 및 예상 범위

MVP1 안정화가 끝나고, 실제 사용자(가상 A기업 시나리오 기준) 피드백 또는 팀 내부 검증을 거친 뒤 착수한다. MVP2 예상 범위(요구사항정의서 기준, 아직 설계 착수 전):

- AI 비서(FR-029~035): 챗봇 호출, 일반 질의응답, 문서 검색, 자연어 일정 조회/생성, 일정 알림
- 대시보드 AI 요약 실기능 전환
- 관리자 CRUD API (FR-001~008) + RBAC 도메인 실제 구현 (`db-schema.md` 5번 "MVP1 미구현" 스키마 활성화)
- 알림 서비스(FR-026~028, 화면 존재 여부 확인 필요 — `product-specs/index.md` 참고)

## 6. 이 로드맵에 반영되지 않은 것

- 구체적인 일정(주 단위 타임라인)은 아직 정하지 않았다. 개인 프로젝트 성격상 완료 목표일보다 "각 단계의 진입 조건 충족"을 기준으로 진행한다.
- MVP2 이후(MVP3+) 범위는 정의하지 않는다. `design-docs/core-beliefs.md` 3번 원칙대로, 당장 필요하지 않은 범위를 미리 설계하지 않는다.
