# SECURITY.md

## 1. 목적과 범위

이 문서는 `flow-bi`의 코드, 데이터, API, 운영 설정, AI 기능에 적용되는 최소 보안 기준을 정의한다. 모든 기능은 보안을 사후 검토 항목이 아니라 설계와 완료 조건의 일부로 취급해야 한다.

## 2. 보호 대상

주요 보호 대상은 다음과 같다.

- 사번, 이름, 이메일, 전화번호, 프로필과 조직 정보
- 비밀번호 해시와 인증 자격정보
- Access Token, Refresh Token과 세션 정보
- 개인·팀·프로젝트 일정 및 참석자 정보
- 회의실 예약 정보
- 역할, 권한, 관리자 기능
- 사내 문서와 AI 대화 내용
- 환경변수, API Key, DB 접속정보와 기타 비밀정보
- 감사 및 보안 로그

## 3. 기본 원칙

- 최소 권한: 사용자와 시스템 구성요소에는 필요한 권한만 부여한다.
- 기본 거부: 명시적으로 허용되지 않은 접근은 거부한다.
- 심층 방어: UI 표시 제한, API 권한 검사, 데이터 제약을 함께 적용한다.
- 데이터 최소화: 업무에 필요한 정보만 수집·저장·노출한다.
- 안전한 실패: 오류 발생 시 권한이나 민감정보가 노출되지 않도록 실패한다.
- 추적 가능성: 중요한 인증·권한·변경 이벤트는 감사 가능해야 한다.

## 4. 인증

### 4.1 로그인

- 사용자는 고유한 사번과 비밀번호로 로그인한다.
- 비밀번호는 평문 또는 복호화 가능한 암호문으로 저장해서는 안 된다.
- 비밀번호는 검증된 단방향 비밀번호 해시 알고리즘과 안전한 파라미터로 저장한다.
- 기본 비밀번호를 발급하는 경우 최초 로그인 후 변경을 강제해야 한다.
- 로그인 실패 응답은 계정 존재 여부를 과도하게 노출하지 않는다.
- 반복 로그인 실패에 대한 제한 정책을 적용해야 한다. 구체적 임계값은 인증 Design Doc에서 정한다.

### 4.2 JWT와 세션

- Access Token은 짧은 만료 시간을 가져야 한다.
- Refresh Token은 Access Token보다 긴 수명을 가지며 서버에서 폐기 가능해야 한다.
- 로그아웃 시 해당 Refresh Token을 무효화한다.
- 전체 로그아웃 또는 관리자 강제 로그아웃을 지원할 수 있도록 사용자별 세션을 식별한다.
- Refresh Token 회전과 재사용 탐지 정책은 인증 Design Doc에서 확정한다.
- 토큰을 URL, 로그, 오류 메시지에 포함하지 않는다.
- 브라우저 저장 위치와 CSRF 방어 방식은 인증 Design Doc에서 함께 결정한다.

현재 ERD의 토큰 저장 구조는 기준선이며 전체 문서 검토 전 임의로 변경하지 않는다. 보안 개선 필요사항은 검토 대기 항목으로 기록한다.

## 5. 인가와 데이터 접근

- 모든 보호 API는 인증된 사용자 신원을 백엔드에서 확인한다.
- 프런트엔드의 메뉴 숨김은 보안 통제가 아니다.
- 관리자 기능은 명시적인 역할 또는 권한이 있는 사용자만 수행할 수 있다.
- 직원·팀·역할·권한 변경은 해당 관리 권한을 요구한다.
- 일정 조회는 일정 유형, 작성자, 참석자, 연결된 팀·프로젝트와 현재 사용자 소속을 기준으로 판정한다.
- 개인 일정은 작성자와 참석자, 팀 일정은 연결된 팀 소속 사용자와 참석자, 프로젝트 일정은 연결된 프로젝트 참여자와 참석자에게만 공개한다.
- 일정 수정·삭제는 소유권과 부여된 권한을 함께 확인한다.
- 회의실 예약 수정·취소는 예약자 또는 허용된 관리자만 수행한다.
- 객체 ID를 바꿔 다른 사용자의 데이터에 접근하는 IDOR를 방지해야 한다.
- 권한 판정 로직은 가능한 한 중앙화하고 기능마다 다르게 해석하지 않는다.

## 6. 입력 검증과 출력 처리

- 클라이언트 입력은 신뢰하지 않으며 백엔드에서 형식, 길이, 범위, 허용값을 검증한다.
- 식별자는 현재 사용자에게 접근 권한이 있는 객체인지 확인한다.
- SQL은 파라미터 바인딩 또는 ORM을 통해 실행하고 문자열 연결로 조립하지 않는다.
- 사용자 입력을 HTML로 출력할 때 XSS를 방지한다.
- 파일 업로드 기능을 도입할 경우 형식, 크기, 저장 위치, 접근 권한과 악성 파일 검사를 정의한다.
- 내부 예외, 스택 트레이스, SQL, 비밀정보를 API 응답에 노출하지 않는다.

## 7. 개인정보와 민감정보

- API는 화면에 필요한 개인정보만 반환한다.
- 목록 응답과 검색 결과에서 민감정보를 불필요하게 포함하지 않는다.
- 로그에 비밀번호, 비밀번호 해시, 전체 토큰, Authorization 헤더를 기록하지 않는다.
- 이메일과 전화번호 등 개인정보의 로그 기록은 마스킹을 기본으로 한다.
- 직원과 팀은 삭제 요청 시 비활성화하고 기존 일정·예약·감사 관계를 보존한다.
- 일정과 예약은 취소 또는 Soft Delete하며 일반 기능에서 물리 삭제하지 않는다.
- 개인정보의 세부 보존 기간과 익명화 정책은 운영 정책에서 별도로 확정한다.
- 개발 및 테스트 환경에서 운영 개인정보를 사용하지 않는다.

## 8. 비밀정보 관리

- 비밀정보를 소스 코드, Markdown, 예제 설정 또는 Git 이력에 커밋하지 않는다.
- 비밀정보는 환경변수 또는 승인된 비밀 저장소를 통해 주입한다.
- `.env` 등 로컬 비밀 파일은 버전 관리에서 제외한다.
- 예제 설정에는 실제 값이 아닌 명확한 플레이스홀더만 사용한다.
- 비밀정보가 노출되면 즉시 작업을 중단하고 사람에게 알리며 해당 값을 폐기·교체한다.

## 9. 데이터베이스와 Redis

- 애플리케이션 DB 계정에는 필요한 최소 권한만 부여한다.
- 데이터 무결성은 애플리케이션 검증과 DB 제약조건을 함께 사용한다.
- 마이그레이션은 리뷰와 검증 없이 운영 데이터에 직접 적용하지 않는다.
- 백업과 복구 정책은 배포 환경 확정 시 정의한다.
- Redis에 비밀번호, 원문 토큰 또는 장기 보존 개인정보를 무기한 저장하지 않는다.
- Redis 키에는 불필요한 개인정보를 포함하지 않는다.
- 캐시 데이터에는 명확한 만료 및 무효화 정책을 둔다.

## 10. 로깅과 감사

다음 이벤트는 사용자, 시간, 대상, 결과를 추적할 수 있어야 한다.

- 로그인 성공·실패와 로그아웃
- 비밀번호 변경 및 세션 무효화
- 역할과 권한의 부여·회수
- 직원 활성화·비활성화
- 중요 일정의 생성·수정·삭제
- 회의실 예약 생성·변경·취소
- AI가 요청하거나 실행한 업무 변경

감사 로그에는 비밀정보와 과도한 개인정보를 포함하지 않는다. 로그 자체의 조회 권한과 보존 기간은 배포·운영 정책에서 확정한다.

## 11. AI 기능 보안

AI 기능은 도입 시 다음 규칙을 MUST 준수한다.

- AI는 사용자의 권한 범위를 넘어 데이터를 조회하지 않는다.
- AI는 DB나 내부 서비스를 직접 조작하지 않고 승인된 애플리케이션 API를 사용한다.
- 일정 생성·수정·삭제, 예약·취소 등 상태 변경은 실행 전 사용자에게 대상과 결과를 확인받는다.
- 문서 검색 결과는 문서 접근 권한을 적용한 뒤 모델에 전달한다.
- 프롬프트 인젝션을 신뢰하지 않으며 검색 문서의 지시를 시스템 명령으로 실행하지 않는다.
- 모델 공급자에게 전송되는 개인정보와 사내정보를 최소화한다.
- AI 응답은 권한이나 비즈니스 검증 결과를 대체하지 않는다.
- 실행된 도구, 입력 요약, 결과와 승인 여부를 감사 가능하게 기록한다.

AI 모델과 문서 검색 방식이 확정되기 전에는 실제 민감정보를 외부 모델에 전송해서는 안 된다.

## 12. 프런트엔드 보안

- 프런트엔드 번들에 비밀정보를 포함하지 않는다.
- 사용자에게 노출되어도 되는 공개 설정만 빌드 변수로 사용한다.
- 인증 상태만으로 권한을 추정하지 않고 서버의 권한 응답을 사용한다.
- 사용자 입력을 임의 HTML로 렌더링하지 않는다.
- 오류 화면에 내부 구현 정보나 민감정보를 표시하지 않는다.
- 의존성 추가 시 유지보수 상태, 라이선스와 알려진 취약점을 검토한다.

## 13. 보안 검증

보안 관련 변경은 최소한 다음을 검증한다.

- 인증되지 않은 요청이 거부되는가
- 권한이 없는 사용자의 요청이 거부되는가
- 다른 사용자의 객체 ID로 접근할 수 없는가
- 잘못된 입력이 안전하게 거부되는가
- 로그와 오류 응답에 민감정보가 없는가
- 토큰 만료, 로그아웃, 무효화가 의도대로 동작하는가
- 동시 요청이 권한 또는 데이터 제약을 우회하지 않는가

자동화 도구가 아직 확정되지 않았더라도 위 시나리오는 기능의 인수 조건에 포함한다.

Harness 실행 중 보안 규칙 또는 시스템 Permission 위반이 발생하면 같은 작업을 자동 재시도하지 않는다. 해당 Task를 즉시 중단하고 위반 내용과 영향을 기록한 뒤 사람의 확인을 요청한다.

## 14. 사람의 승인이 필요한 보안 변경

- 인증 방식 또는 토큰 저장 방식 변경
- 역할·권한 모델 변경
- 개인정보 수집·보관·공개 범위 변경
- 외부 AI 또는 문서 검색 서비스로 데이터 전송
- CORS, CSRF, 쿠키 또는 세션 정책 완화
- 보안 검증의 비활성화
- 암호화·해시 알고리즘 또는 보안 라이브러리 변경

## 15. 보안 사고 대응

비밀정보 노출, 권한 우회, 개인정보 유출 또는 데이터 손상 가능성을 발견하면 다음을 수행한다.

1. 추가 노출이나 변경을 중단한다.
2. 발견 사실과 영향 범위를 사람에게 알린다.
3. 로그나 증거를 임의 삭제하지 않는다.
4. 비밀정보는 승인된 절차로 폐기하고 교체한다.
5. 수정 후 회귀 검증과 원인 기록을 남긴다.
