# SECURITY.md

이 문서는 인증/인가, 토큰 정책, 민감정보 처리 등 보안 관련 결정을 다룬다. `exec-plans/tech-debt-tracker.md`에 "상" 심각도로 남아있던 **access/refresh token 만료시간을 이 문서에서 확정**한다.

---

## 1. 기본 원칙 (NFR-001, NFR-002)

- 로그인한 사용자만 서비스를 이용할 수 있다 (NFR-001) — `auth`/`login`/`refresh` 관련 엔드포인트를 제외한 모든 API는 `JwtAuthenticationFilter`로 보호한다 (ARCHITECTURE.md 3.3).
- 비밀번호는 암호화하여 저장한다 (NFR-002) — BCrypt 해시 사용, 평문 저장/로깅 절대 금지.

## 2. 토큰 정책 (확정 — 기존 미정 사항 해소)

| 항목 | 값 | 근거 |
|---|---|---|
| Access token 만료시간 | **30분** | 짧게 유지해 탈취 시 피해 범위를 최소화. Redis 블랙리스트가 없는 MVP1 구조(`ARCHITECTURE.md` 6번)에서 로그아웃 후에도 access token 자체는 만료 전까지 유효하므로, 만료시간을 짧게 잡는 것이 유일한 방어선이다. |
| Refresh token 만료시간 | **14일** | 사내 도구 특성상 매일 재로그인은 사용성을 해친다. 2주 정도면 보안과 편의의 균형점으로 적절하다고 판단. |
| Refresh 로테이션 | 매 refresh 요청마다 새 refresh token 발급, 기존 `user_tokens` 레코드는 새 토큰으로 갱신(같은 device 레코드를 업데이트) | 탈취된 refresh token이 계속 재사용되는 것을 방지 |
| Refresh token 재사용 탐지 | 이미 사용되어 무효화된 refresh token으로 재발급을 시도하면, 해당 사용자의 **모든 device**의 `user_tokens`를 삭제하고 전체 재로그인을 요구 | 탈취 의심 시나리오에 대한 대응. 정상적인 사용자라면 이미 로테이션된 토큰을 다시 쓸 일이 없으므로, 재사용 시도 자체가 이상 신호 |

> 이 값들은 초기 제안값이다. 실사용 후 30분이 너무 잦은 재인증을 유발하면 조정 가능하나, 조정 시 이 문서와 `exec-plans/active/auth-domain.md`를 함께 갱신한다.

## 3. 비밀번호 정책

- 해시: BCrypt (strength 10 이상)
- 초기 비밀번호: 시딩 시 임직원별로 임의 생성하거나 공통 초기값 부여 후 로그인 화면에서 변경 유도 (`login.md`의 "비밀번호 변경" 미해결 이슈와 연결 — 최초 로그인 강제 변경은 하지 않기로 가정했으므로, 초기 비밀번호 강도 자체가 낮으면 위험. 시딩 시 임직원별 고유값 생성을 권장)
- 최소 길이/복잡도 규칙: 아직 명시된 요구사항 없음. 최소 8자 이상을 기본으로 제안 (구현 시 확정)

## 4. 입력 검증

- 모든 요청 DTO에 Bean Validation(`@NotNull`, `@Size`, `@Email` 등) 적용, 컨트롤러에서 `@Valid`로 검증 (springdoc이 이 어노테이션을 문서에도 자동 반영, `references/springdoc-openapi-llms.txt` 참고)
- JPA/Spring Data 사용으로 SQL Injection은 기본적으로 방어됨 (native query를 쓸 경우 반드시 파라미터 바인딩 사용, 문자열 concat 금지)
- 프론트엔드에서 사용자 입력을 그대로 HTML로 렌더링하는 지점(예: 일정 상세 설명)이 있다면 XSS 방지를 위해 React의 기본 이스케이프를 유지하고 `dangerouslySetInnerHTML` 사용을 금지한다

## 5. 권한 검증 (Authorization)

- 일정 수정/삭제: `schedules.creator_id`가 요청자 본인인지 서비스 레이어에서 검증 (`schedule.md` 참고)
- 마이페이지 수정: 본인 `user_id`에 대해서만 `PATCH /users/me` 허용, URL에 다른 사용자 ID를 넣어도 본인 것만 수정되도록 서버에서 강제 (요청 파라미터의 ID를 신뢰하지 않고, 인증 토큰에서 추출한 사용자 ID를 사용)
- 회의실 예약 수정/취소: 예약자 본인만 가능한지, 아니면 같은 팀 누구나 가능한지는 명시된 요구사항이 없다 — **가정(확인 필요)**: 예약과 연동된 `schedules.creator_id` 기준으로 본인만 가능하게 시작하고, 필요시 확장한다.
- 관리자 권한 체계(RBAC)는 MVP1에 없다(`db-schema.md` 5번). 따라서 MVP1에는 "일반 사용자"만 존재하고 권한 분기 로직 자체가 없다 — 이는 의도된 단순화이지 누락이 아니다.

## 6. 민감정보 처리

- 응답 DTO에 `password_hash`, `refresh_token` 원문을 절대 포함하지 않는다 (AGENTS.md 5.2와 동일)
- 로그에 비밀번호, 토큰 원문, 개인 전화번호/이메일 전체를 남기지 않는다 (RELIABILITY.md 5번과 연결). 필요 시 마스킹 처리.
- 에러 응답(`error.details`)에 내부 스택트레이스나 SQL 쿼리를 노출하지 않는다 — 사용자에게는 이해 가능한 메시지만, 상세 원인은 서버 로그에만 남긴다.

## 7. 전송 보안 및 CORS

- HTTPS 적용 여부는 배포 환경에 종속되며 아직 미정이다 (`ARCHITECTURE.md` 8번, `RELIABILITY.md` 6번과 연결). 배포 환경이 정해지면 이 문서에 TLS 종단 지점을 명시한다.
- 프론트엔드와 백엔드가 다른 origin에서 서빙된다면 CORS 설정이 필요하다. 모노레포이지만 배포 시 도메인이 분리될 수 있으므로, 허용 origin을 화이트리스트로 명시적으로 관리한다(와일드카드 `*` 금지, 특히 인증 정보를 포함하는 요청이므로).

## 8. 로그인 시도 제한 (권장, 아직 미구현 항목)

무차별 대입 공격(brute force) 방지를 위해 동일 사번에 대해 짧은 시간 내 반복 로그인 실패 시 일시적으로 잠그는 정책을 권장한다. 구체적인 임계치(예: 5회 실패 시 5분 잠금)는 아직 정하지 않았다 — MVP1 필수 요구사항은 아니므로, 시간이 부족하면 MVP1 안정화 단계로 미뤄도 무방하다(`PLANS.md` 3번 진입 조건과 연결해 판단).

## 9. 남은 미정 사항

- HTTPS/TLS 적용 방식 (배포 환경 결정 종속)
- 로그인 시도 제한 임계치 및 구현 여부
- 비밀번호 최소 복잡도 규칙 최종 확정
- 회의실 예약 수정/취소 권한 범위(본인만 vs 팀 전체) 최종 확인
