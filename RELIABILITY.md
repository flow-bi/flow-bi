# RELIABILITY.md

이 문서는 서비스가 예상대로 계속 동작하게 만드는 기준 — 데이터 정합성, 동시성, 장애 대응 — 을 다룬다. 배포/인프라 자체(어디서 얼마나 가용하게 띄울지)는 아직 미정이므로(`ARCHITECTURE.md` 8번), 이 문서는 인프라와 무관하게 **애플리케이션 레벨에서 챙겨야 할 신뢰성**에 집중한다.

---

## 1. 가용성 목표

NFR-005(24시간 이용 가능)는 명시되어 있으나, 이를 뒷받침할 배포 환경/이중화 여부가 아직 결정되지 않았다(`ARCHITECTURE.md` 8번 "미정 사항"). 애플리케이션 코드 레벨에서는 최소한 다음을 지킨다:

- 헬스체크 엔드포인트 제공 (구체 경로는 배포 방식 결정 시 정함, Spring Boot Actuator 활용 고려)
- 단일 요청 실패가 서버 전체를 죽이지 않도록 `GlobalExceptionHandler`로 모든 예외를 잡아 Envelope 에러 응답으로 변환 (AGENTS.md 5.1)

## 2. 데이터 정합성 — 도메인 간 연동 지점

이 프로젝트에서 정합성이 깨지기 가장 쉬운 지점은 **회의실 예약과 일정이 두 도메인(`meetingroom`, `schedule`)에 걸쳐 함께 생성/삭제**되는 부분이다(`meeting-room.md`, `db-schema.md` 참고).

### 원자성 보장 방식

`meetingroom` 도메인의 예약 생성/취소 서비스가 `schedule` 도메인의 서비스를 호출할 때, 두 작업(예약 레코드 생성/삭제 + 일정 생성/삭제)은 **하나의 트랜잭션**으로 묶여야 한다. 모놀리식 구조(단일 DB, 단일 애플리케이션)이므로 별도의 분산 트랜잭션(Saga 등) 없이, `@Transactional`로 예약 서비스 메서드 전체를 감싸는 것으로 충분하다.

```
@Transactional
meetingroom.reservationService.create(...):
    schedule = scheduleService.createForReservation(...)  // 같은 트랜잭션 내
    reservation = reservationRepository.save(...)
    // 둘 중 하나라도 실패하면 전체 롤백
```

이렇게 하지 않으면 "예약은 생성됐는데 일정은 없는" 또는 그 반대의 불일치 상태가 발생할 수 있다. `exec-plans/active/meetingroom-domain.md`의 Service 체크리스트 항목에 이 트랜잭션 경계를 명시적으로 포함시킬 것.

### 취소 시에도 동일 원칙 적용

`meeting-room.md`에서 확정한 대로 예약 취소 시 연동된 `schedules`를 삭제하는데, 이 역시 같은 트랜잭션 안에서 처리해 "예약은 취소됐는데 캘린더에는 일정이 남아있는" 상태를 막는다.

## 3. 동시성 — 회의실 예약 레이스 컨디션

`db-schema.md`에서 확정한 대로 시간 겹침 검증은 DB 제약이 아닌 서비스 레이어에서 처리한다. 이 방식의 약점은 **두 사용자가 거의 동시에 같은 시간대를 예약 요청하면, 둘 다 겹침 검사를 통과해버릴 수 있다**는 것이다(조회 후 삽입 사이의 간극, TOCTOU).

**대응 방안(구현 시 선택)**:
- **비관적 락**: 예약 생성 트랜잭션에서 해당 `room_id`에 대한 락을 잡고(`SELECT ... FOR UPDATE` 등) 겹침 검사와 삽입을 하나의 락 구간 안에서 처리
- **DB 유니크 제약 보조 장치**: 완전한 겹침 방지는 아니더라도, 최소한 동일 `room_id + start_at` 조합에 대한 유니크 제약을 추가해 극단적인 동시 요청을 걸러내는 안전망 추가 (선택, 완벽한 해결책은 아님)

MVP1에서는 최소한 비관적 락 적용을 권장하며, 부하가 실제로 문제가 될 정도로 커지면 그때 낙관적 락이나 큐잉 방식을 재검토한다(`design-docs/core-beliefs.md` 7번 원칙과 동일).

## 4. 인증 관련 신뢰성

- Refresh token 로테이션 시, 이전 토큰이 이미 사용된 뒤 다시 사용되는 경우(탈취 의심 시나리오)를 어떻게 처리할지는 `SECURITY.md`에서 다룬다.
- Access token 만료 직후 요청이 실패하면, 프론트엔드는 자동으로 refresh를 시도한 뒤 원래 요청을 재시도한다(`FRONTEND.md` 4번의 `apiFetch` 401 처리와 연결). 재시도는 1회만 하고, 재시도도 실패하면 로그인 페이지로 리다이렉트한다(무한 재시도 루프 방지).

## 5. 로깅/모니터링

아직 결정되지 않았다(`ARCHITECTURE.md` 8번 "테스트 전략 미정"과 함께 배포 환경 결정 시 같이 다룰 것). 최소한 다음은 코드 레벨에서 지금부터 지킨다:

- `GlobalExceptionHandler`에서 잡히는 모든 예외는 서버 로그에 스택트레이스와 함께 남긴다 (사용자에게는 Envelope 에러 메시지만 노출, 내부 스택트레이스는 노출 금지 — `SECURITY.md`와 연결)
- 비밀번호, 토큰 원문은 어떤 로그에도 남기지 않는다 (AGENTS.md 7번과 동일)

## 6. 남은 미정 사항

- 로깅/모니터링 도구 선택 (Sentry, CloudWatch 등)
- 백업/복구 정책 (PostgreSQL 백업 주기 등 — 아직 논의된 적 없음, 배포 환경 결정과 함께 정할 것)
