# FRONTEND.md

## 1. 문서 상태

- 상태: Initial Draft
- 범위: 프런트엔드 아키텍처와 구현 기준
- 미확정: 빌드 도구, 라우터, 테스트 도구, 캘린더 라이브러리, 날짜 라이브러리, API 생성 방식

미확정 기술은 기능 구현의 필요만으로 임의 선택하지 않는다. Design Doc 또는 Active Plan에서 근거와 검증 방법을 정하고 승인을 받은 뒤 도입한다.

## 2. 기술 스택

| 구분 | 기술 |
| --- | --- |
| UI | React + TypeScript |
| 서버 상태 | TanStack Query |
| 클라이언트 전역 상태 | Zustand |
| Form | React Hook Form |
| Schema 검증 | Zod |
| Styling | Tailwind CSS |
| Package Manager | npm |

## 3. 책임

프런트엔드는 다음을 담당한다.

- 인증된 사용자에게 그룹웨어 화면 제공
- 대시보드, 일정, 조직도, 회의실, 관리, 마이페이지 UI
- 사용자의 입력과 상호작용 처리
- API 요청·응답의 화면 상태 표현
- 반응형 레이아웃과 접근성
- AI 비서 화면의 추후 확장 지점 제공

프런트엔드는 다음의 최종 책임자가 아니다.

- 인증과 권한 판정
- 비즈니스 규칙과 데이터 정합성
- 회의실 중복 예약 판정
- 일정 공개 범위의 최종 판정
- AI 도구 실행의 승인과 서버 검증

## 4. 권장 소스 구조

```text
frontend/
├── src/
│   ├── app/          # 앱 초기화, Provider, Router, 전역 설정
│   ├── pages/        # 라우트 단위 화면 조립
│   ├── features/     # 기능별 UI, 상태, API 연결
│   ├── entities/     # 안정된 핵심 도메인 표현
│   ├── shared/       # 범용 UI, 유틸리티, API 기반 코드
│   └── assets/       # 정적 자산
└── tests/            # 공통 테스트 설정이 필요한 경우
```

세부 구조 원칙:

- 기능 코드는 `features/{feature-name}/` 안에서 응집시킨다.
- `pages/`는 기능 컴포넌트를 조립하고 라우트 경계를 담당한다.
- `shared/`는 특정 도메인 의미를 가져서는 안 된다.
- 기능 간 내부 파일을 직접 참조하지 않고 공개 진입점을 사용한다.
- 순환 의존을 허용하지 않는다.
- 초기에는 필요 이상의 계층과 빈 디렉터리를 만들지 않는다.

## 5. 기능 모듈 후보

현재 요구사항에 따른 기능 경계는 다음과 같다.

- `auth`: 로그인, 로그아웃, 비밀번호 변경
- `dashboard`: 오늘 일정, 주간 일정, 팀원 상태 요약
- `calendar`: 월간·주간·일간 조회, 일정 상세와 편집
- `organization`: 조직도와 직원 정보
- `rooms`: 회의실 검색, 현황, 예약
- `admin`: 직원·팀 관리
- `profile`: 개인정보와 업무 상태 관리
- `notifications`: 알림 설정과 알림 표시; 추후 구현
- `assistant`: AI 채팅과 단축어; 추후 구현
- `theme`: 사용자 테마 선택

기능 경계는 실제 Product Spec과 API가 확정될 때 조정할 수 있다.

## 6. 상태 소유권

| 상태 유형 | 소유 도구 | 예시 |
| --- | --- | --- |
| 서버 상태 | TanStack Query | 일정, 직원, 회의실, 예약 |
| 폼 상태 | React Hook Form | 로그인, 일정 생성, 예약 |
| 공유 UI 상태 | Zustand | 사용자 테마, 전역 패널 상태 |
| 로컬 UI 상태 | React state | 선택된 탭, 모달 내부 단계 |
| URL 상태 | Router | 조회 날짜, 캘린더 보기, 검색 조건 |

- 서버에서 가져오는 데이터와 캐시는 TanStack Query가 관리한다.
- 여러 화면에서 공유하는 순수 클라이언트 상태는 Zustand가 관리한다.
- 로컬 UI 상태는 가능한 컴포넌트 가까이에 둔다.
- 폼 상태와 검증은 React Hook Form과 Zod를 사용한다.
- 서버의 권한 판정을 프런트엔드 상태로 대체하지 않는다.

조회 조건이 공유·북마크·뒤로 가기의 대상이면 가능한 한 URL 상태로 표현한다.

## 7. 서버 상태 규칙

- Query Key는 기능별 Factory 또는 일관된 생성 규칙으로 관리한다.
- 조회 범위, 사용자 권한, 날짜와 검색 조건을 Query Key에 반영한다.
- Mutation 성공 후 영향받는 Query만 갱신 또는 무효화한다.
- 오래된 데이터 유지 여부와 재조회 정책을 기능 특성에 맞게 정한다.
- 민감한 사용자별 캐시는 로그아웃 시 제거한다.
- 낙관적 갱신은 되돌리기와 충돌 처리가 명확한 기능에만 사용한다.

## 8. API 계층

기능별 API 모듈은 다음 책임을 가진다.

- URL, HTTP 메서드와 파라미터 구성
- API 요청·응답 Type 연결
- 네트워크 오류를 공통 오류 모델로 변환
- 인증 헤더 또는 쿠키 정책 적용

UI 컴포넌트는 원시 HTTP 세부사항에 직접 의존하지 않는다. 응답 DTO를 화면에 적합한 모델로 변환할 수 있지만 원본 의미를 바꾸지 않는다.

## 9. 화면별 기능 기준

### 9.1 로그인

- 사번과 비밀번호 입력
- 로그인 오류 표시
- 기본 비밀번호 사용자의 비밀번호 변경 흐름
- 인증 성공 후 허용된 초기 화면으로 이동

### 9.2 대시보드

- 오늘 일정
- 주간 일정
- 팀원 업무 상태
- AI 오늘 일정 요약은 AI 기능 도입 전까지 확장 영역으로 유지

### 9.3 캘린더

- 초기 화면은 월간 보기
- 월간·주간·일간 보기 전환
- 날짜 선택 시 일간 일정 패널 표시
- 일정 선택 시 상세 모달 표시
- 일정 추가
- 일정 유형은 개인·팀·프로젝트 중 정확히 하나만 선택
- 팀 유형은 여러 팀, 프로젝트 유형은 여러 프로젝트를 선택할 수 있음
- 모든 유형에서 여러 참석자를 선택할 수 있음
- 개인 일정은 작성자·참석자, 팀 일정은 연결된 팀·참석자, 프로젝트 일정은 연결된 프로젝트·참석자에게 공개됨을 안내
- 공개 대상은 유형에서 파생되며 유형과 모순되는 공개 범위를 별도로 선택하지 않음

### 9.4 조직도

- 계층형 팀 구조 표시
- 직원 정보와 업무 상태 표시
- 사람 선택 후 채팅 연결은 확장 범위

### 9.5 회의실

- 회의실 목록 및 시설 정보
- 09:00~18:00 예약 현황 타임라인
- 수용 인원, 날짜, 시간대, 상태 기반 검색
- 검색 결과를 적합도 순으로 표시할 수 있는 확장 구조
- 선택한 회의실의 예약 패널

검색 순위 계산 기준은 아직 미확정이며 Product Spec 또는 Design Doc 없이 임의 구현하지 않는다.

### 9.6 관리

- 관리자 권한이 있는 사용자에게만 진입 UI 제공
- 직원과 팀 목록, 수정과 비활성화 UI
- 비활성화 전 영향과 기존 관계가 보존됨을 확인
- 실제 권한 판정은 서버 응답을 따른다.

### 9.7 마이페이지

- 변경 불가: 프로필 사진, 이름, 사번, 직급·직책
- 변경 가능: 이메일, 사내 전화번호, 업무 상태

## 10. 오류 및 비동기 상태

모든 데이터 화면은 적용 가능한 다음 상태를 가진다.

- 초기 로딩
- 부분 또는 재조회 로딩
- 정상 데이터
- 빈 결과
- 입력 검증 오류
- 인증 만료
- 권한 거부
- 데이터 충돌
- 서버 또는 네트워크 오류

오류 발생 시 기존에 유효한 데이터를 무조건 제거하지 않는다. 재시도 가능한 오류에는 재시도 수단을 제공한다.

## 11. 성능 기준

- 캘린더는 현재 조회 범위에 필요한 데이터만 요청한다.
- 큰 목록은 페이지네이션, 가상화 또는 제한된 검색을 검토한다.
- 서버 상태를 불필요하게 중복 요청하지 않는다.
- 무거운 화면과 확장 기능은 필요할 때 지연 로딩을 검토한다.
- 성능 최적화는 측정 결과와 사용자 영향을 근거로 수행한다.

## 12. 테스트 전략

테스트 도구는 아직 미확정이지만 다음 계층을 목표로 한다.

- 순수 함수와 Schema 단위 테스트
- Form과 기능 컴포넌트의 사용자 동작 테스트
- API 경계를 통제한 화면 통합 테스트
- 로그인, 일정 생성, 회의실 예약 등 핵심 흐름의 E2E 테스트

테스트는 구현 세부사항보다 사용자의 관찰 가능한 동작을 검증한다. 구체적인 도구는 ADR 또는 승인된 Active Plan으로 결정한다.
