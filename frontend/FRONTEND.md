# FRONTEND.md

## 1. 문서 상태

- 상태: Initial Draft
- 범위: 프런트엔드 아키텍처와 구현 기준
- 미확정: 빌드 도구, 라우터, 테스트 도구, 날짜 라이브러리, API 생성 방식

미확정 기술은 기능 구현의 필요만으로 임의 선택하지 않는다. Design Doc 또는 Active Plan에서 근거와 검증 방법을 정하고 승인을 받은 뒤 도입한다.

이 문서는 기능별 요구사항을 정의하거나 중복하지 않는다. 기능의 목적, 사용자 시나리오, 업무 규칙, 권한과 인수 조건은 `product-specs/`의 승인된 Product Spec을 기준으로 한다.

## 2. 기술 스택

| 구분 | 기술 |
| --- | --- |
| UI | React + TypeScript |
| 서버 상태 | TanStack Query |
| 클라이언트 전역 상태 | Zustand |
| Form | React Hook Form |
| Schema 검증 | Zod |
| Styling | Tailwind CSS |
| Package Manager | npm |

## 3. 책임

프런트엔드는 다음을 담당한다.

- 인증된 사용자에게 그룹웨어 화면 제공
- 승인된 Product Spec의 기능 요구사항을 화면과 사용자 상호작용으로 구현
- 사용자의 입력과 상호작용 처리
- API 요청·응답의 화면 상태 표현
- 반응형 레이아웃과 접근성

프런트엔드는 다음의 최종 책임자가 아니다.

- 인증과 권한 판정
- 비즈니스 규칙과 데이터 정합성
- 동시성에 민감한 작업의 충돌 판정
- 데이터 공개 범위의 최종 판정
- 외부 도구 또는 자동화된 변경 작업의 승인과 서버 검증

## 4. 권장 소스 구조

```text
frontend/
├── src/
│   ├── app/          # 앱 초기화, Provider, Router, 전역 설정
│   ├── pages/        # 라우트 단위 화면 조립
│   ├── features/     # 기능별 UI, 상태, API 연결
│   ├── entities/     # 안정된 핵심 도메인 표현
│   ├── shared/       # 범용 UI, 유틸리티, API 기반 코드
│   └── assets/       # 정적 자산
└── tests/            # 공통 테스트 설정이 필요한 경우
```

세부 구조 원칙:

- 기능 코드는 `features/{feature-name}/` 안에서 응집시킨다.
- `pages/`는 기능 컴포넌트를 조립하고 라우트 경계를 담당한다.
- `shared/`는 특정 도메인 의미를 가져서는 안 된다.
- 기능 간 내부 파일을 직접 참조하지 않고 공개 진입점을 사용한다.
- 순환 의존을 허용하지 않는다.
- 초기에는 필요 이상의 계층과 빈 디렉터리를 만들지 않는다.

## 5. 기능 모듈 경계

- 기능 모듈의 목록과 범위는 승인된 Product Spec과 `ARCHITECTURE.md`의 도메인 경계를 기준으로 정한다.
- 화면 이름만으로 모듈을 나누지 않고 응집된 사용자 기능과 변경 이유를 기준으로 나눈다.
- Product Spec이나 API 계약이 확정되지 않은 기능 경계를 이 문서에서 선행 확정하지 않는다.
- 기능 경계의 신설·분리·통합이 상위 도메인 경계나 의존 방향을 바꾸면 Design Doc 또는 ADR과 사람의 승인을 먼저 받는다.

## 6. 상태 소유권

| 상태 유형 | 소유 도구 | 예시 |
| --- | --- | --- |
| 서버 상태 | TanStack Query | API에서 조회한 목록과 상세 데이터 |
| 폼 상태 | React Hook Form | 사용자 입력과 제출 상태 |
| 공유 UI 상태 | Zustand | 사용자 테마, 전역 패널 상태 |
| 로컬 UI 상태 | React state | 선택된 탭, 모달 내부 단계 |
| URL 상태 | Router | 조회 범위, 보기 방식, 검색 조건 |

- 서버에서 가져오는 데이터와 캐시는 TanStack Query가 관리한다.
- 여러 화면에서 공유하는 순수 클라이언트 상태는 Zustand가 관리한다.
- 로컬 UI 상태는 가능한 컴포넌트 가까이에 둔다.
- 폼 상태와 검증은 React Hook Form과 Zod를 사용한다.
- 서버의 권한 판정을 프런트엔드 상태로 대체하지 않는다.

조회 조건이 공유·북마크·뒤로 가기의 대상이면 가능한 한 URL 상태로 표현한다.

## 7. 서버 상태 규칙

- Query Key는 기능별 Factory 또는 일관된 생성 규칙으로 관리한다.
- 조회 범위, 사용자 권한, 날짜와 검색 조건을 Query Key에 반영한다.
- Mutation 성공 후 영향받는 Query만 갱신 또는 무효화한다.
- 오래된 데이터 유지 여부와 재조회 정책을 기능 특성에 맞게 정한다.
- 민감한 사용자별 캐시는 로그아웃 시 제거한다.
- 낙관적 갱신은 되돌리기와 충돌 처리가 명확한 기능에만 사용한다.

## 8. API 계층

기능별 API 모듈은 다음 책임을 가진다.

- URL, HTTP 메서드와 파라미터 구성
- API 요청·응답 Type 연결
- 네트워크 오류를 공통 오류 모델로 변환
- 인증 헤더 또는 쿠키 정책 적용

UI 컴포넌트는 원시 HTTP 세부사항에 직접 의존하지 않는다. 응답 DTO를 화면에 적합한 모델로 변환할 수 있지만 원본 의미를 바꾸지 않는다.

## 9. 기능 요구사항 참조

- 화면별 기능 범위, 사용자 시나리오, 업무 규칙, 권한과 인수 조건은 `product-specs/`의 해당 Product Spec을 따른다.
- Product Spec의 요구사항을 이 문서에 다시 작성하지 않는다.
- Product Spec을 프런트엔드 구조와 상호작용으로 구체화해야 하면 기능별 Design Doc과 Active Plan에 설계 및 검증 방법을 기록한다.
- Product Spec, Design Doc, API 계약이 서로 충돌하면 임의로 해석하지 않고 구현 전에 충돌을 기록하고 확인받는다.

## 10. 오류 및 비동기 상태

모든 데이터 화면은 적용 가능한 다음 상태를 가진다.

- 초기 로딩
- 부분 또는 재조회 로딩
- 정상 데이터
- 빈 결과
- 입력 검증 오류
- 인증 만료
- 권한 거부
- 데이터 충돌
- 서버 또는 네트워크 오류

오류 발생 시 기존에 유효한 데이터를 무조건 제거하지 않는다. 재시도 가능한 오류에는 재시도 수단을 제공한다.

## 11. 성능 기준

- 데이터 화면은 현재 조회 범위에 필요한 데이터만 요청한다.
- 큰 목록은 페이지네이션, 가상화 또는 제한된 검색을 검토한다.
- 서버 상태를 불필요하게 중복 요청하지 않는다.
- 무거운 화면과 확장 기능은 필요할 때 지연 로딩을 검토한다.
- 성능 최적화는 측정 결과와 사용자 영향을 근거로 수행한다.

## 12. 테스트 전략

테스트 도구는 아직 미확정이지만 다음 계층을 목표로 한다.

- 순수 함수와 Schema 단위 테스트
- Form과 기능 컴포넌트의 사용자 동작 테스트
- API 경계를 통제한 화면 통합 테스트
- Product Spec에서 정한 핵심 사용자 흐름의 E2E 테스트

테스트는 구현 세부사항보다 사용자의 관찰 가능한 동작을 검증한다. 구체적인 도구는 ADR 또는 승인된 Active Plan으로 결정한다.

## 13. 미결정 사항

- 빌드 도구와 개발 서버
- 라우터
- 테스트 및 E2E 도구
- 날짜 처리 라이브러리
- API Client 및 OpenAPI 코드 생성 여부
- 인증 Token의 브라우저 저장 방식
- 지원 브라우저와 최소 화면 너비
