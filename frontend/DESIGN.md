# DESIGN.md

## 1. 문서 상태와 목적

- 상태: Initial Draft
- 목적: `flow-bi` 프런트엔드의 공통 UI·UX·반응형·접근성 기준 정의
- 기준 화면: PC와 모바일 웹

이 문서는 기능 요구사항을 시각적·상호작용 규칙으로 구체화한다. 확정된 디자인 시안이나 디자인 토큰이 없는 항목은 임의로 고정하지 않는다.

## 2. 디자인 원칙

- 사용자가 현재 위치와 다음 행동을 빠르게 이해할 수 있어야 한다.
- 일정과 예약처럼 시간에 민감한 정보는 한눈에 비교할 수 있어야 한다.
- 중요한 상태와 오류는 색상만으로 표현하지 않는다.
- 파괴적 행동은 일반 행동과 시각적으로 구분하고 확인 절차를 제공한다.
- PC와 모바일에서 핵심 기능을 사용할 수 있어야 한다.
- AI가 생성한 내용과 실제 시스템 데이터 또는 실행 결과를 구분한다.

## 3. 테마

초기 테마 후보는 다음과 같다.

### Sky Theme

- 기본 배경 후보: `#EDF1F6`
- 흰색 Surface와 하늘색 계열 강조색 조합

### Purple Theme

- 대표 강조색 후보: `#5541A4`
- 흰색 Surface와 연보라 계열 보조색 조합

색상 후보는 디자인 토큰 확정 전 참고값이다. 배경과 글자, 버튼과 상태 색상은 접근 가능한 명암 대비를 검증한 후 확정한다.

사용자 테마 선택은 색상 토큰만 교체해야 하며 컴포넌트 구조나 의미를 바꾸지 않는다.

## 4. 디자인 토큰

Tailwind CSS 설정에서 다음 의미 기반 토큰을 관리하는 방향을 사용한다.

- `background`: 앱 전체 배경
- `surface`: 카드, 모달, 패널 배경
- `primary`: 주요 행동과 선택 상태
- `secondary`: 보조 행동
- `text-primary`, `text-secondary`, `text-disabled`
- `border`
- `success`, `warning`, `danger`, `info`
- `focus-ring`

컴포넌트에서 임의 Hex 값을 반복하지 않는다. 토큰 이름은 색상 자체보다 역할을 표현해야 한다.

## 5. 전체 레이아웃 (todo: 다함께 점검 한번 필요/왜냐면 이대로 구현되지 않을까?)

### Header

- 좌측: 기업명 또는 서비스 식별자
- 우측: 사용자 이름과 계정 메뉴
- 검색창은 목적과 검색 대상이 확정될 때까지 구현 범위에서 제외

### Navigation

다음 주요 영역으로 이동할 수 있어야 한다.

- 대시보드
- 캘린더
- 조직도
- 회의실
- 관리: 권한이 있을 때만 표시
- 마이페이지
- AI 비서: 기능 활성화 시 표시

현재 위치를 시각적 표시와 접근성 속성으로 전달한다.

### Content

- 화면 제목, 주요 행동, 보조 필터, 본문 순서로 정보 위계를 유지한다.
- 핵심 행동은 화면마다 과도하게 여러 개 강조하지 않는다.
- 목록과 상세 패널은 선택 관계가 명확해야 한다.

## 6. 반응형 원칙

정확한 Breakpoint는 Tailwind 설정 확정 시 결정한다.

- Desktop: Navigation과 본문, 필요 시 우측 패널을 동시에 표시
- Tablet: 본문을 우선하고 보조 패널을 Overlay 또는 접이식으로 전환
- Mobile: 한 번에 하나의 주요 작업을 보여주고 패널과 모달을 전체 화면에 가깝게 표시
- 가로 스크롤이 필요한 회의실 타임라인은 시간축과 회의실 식별 정보를 유지한다. (todo: 가로가 좋기는 한데 화면 크기에 따라서 스크롤이 생기면 불편하지 않을까..!)
- 모바일에서도 일정 생성, 조회, 회의실 예약과 취소가 가능해야 한다.

## 7. 공통 컴포넌트 규칙

### Button

- Primary, Secondary, Tertiary, Danger의 의미를 구분한다.
- 아이콘만 사용하는 Button에는 접근 가능한 이름을 제공한다.
- 비동기 처리 중 상태를 표시하고 중복 제출을 막는다.

### Form Field

- Label, 입력, 도움말, 오류 메시지의 연결을 유지한다.
- 필수 여부를 시각적으로만 표시하지 않는다.
- 비활성화와 읽기 전용 상태를 구분한다.

### Modal

- 일정 상세·생성과 같이 집중된 작업에 사용한다.
- 제목과 명확한 닫기 수단을 제공한다.
- 열린 동안 포커스를 관리하고 닫힌 뒤 원래 위치로 돌려보낸다.
- 저장되지 않은 입력이 있으면 닫기 전에 확인한다.

### Side Panel

- 날짜별 일정, 회의실 예약처럼 현재 화면의 문맥을 유지해야 하는 상세 작업에 사용한다.
- Desktop에서는 우측에 표시하고 작은 화면에서는 Overlay 또는 전체 화면으로 전환한다.
- 선택 대상과 패널 제목을 명확히 연결한다.

### Status

- 텍스트, 아이콘과 색상을 함께 사용한다.
- 동일한 상태는 모든 화면에서 같은 표현을 사용한다.

## 8. 화면별 설계 기준

기능 범위와 구현 요구사항은 FRONTEND.md의 화면별 기능 기준을 따른다. 아래 항목은 같은 기능을 사용자에게 어떻게 보여주고 상호작용할지에 대한 설계 기준이다.

### 8.1 로그인

- 인증 흐름은 사번, 비밀번호, 오류 안내, 비밀번호 변경 유도까지 한 흐름으로 구성한다.
- 인증 실패와 비밀번호 변경 필요 상태는 자격정보 노출 없이 명확히 안내한다.

### 8.2 대시보드

- 정보 우선순위는 AI 요약(확장 영역), 오늘 일정, 주간 일정, 팀원 상태 순으로 유지한다.
- 카드와 목록은 다음 행동과 현재 상태를 빠르게 파악할 수 있게 구성한다.

### 8.3 AI 채팅

- AI 기능이 아직 활성화되지 않은 경우 빈 상태를 실제 동작처럼 보이지 않게 표현한다.
- AI 답변, 근거, 실행 결과는 시각적으로 구분하고 변경 행동은 확인 단계를 제공한다.

### 8.4 캘린더

- 기본 보기와 전환 UI는 월간·주간·일간 흐름을 쉽게 이해하도록 구성한다.
- 일정 유형과 공개 범위는 색상만이 아니라 텍스트·아이콘으로도 식별 가능하게 표현한다.
- 날짜 선택과 일정 선택 시 상세 컨텍스트를 유지하는 패널·모달 흐름을 사용한다.

### 8.5 조직도

- 팀 계층과 선택 상태를 명확히 전달하고, 직원 정보는 권한 범위에 맞게 표시한다.
- 개인정보 노출은 최소화하고 필요한 범위에서만 제공한다.

### 8.6 회의실

- 회의실 목록과 예약 현황은 가독성을 유지하는 정보 우선순위로 구성한다.
- 예약 블록은 시간·제목·소속을 한눈에 볼 수 있게 표시하고, 검색 조건과 결과 정렬 이유를 명확히 안내한다.

### 8.7 관리

- 직원과 팀 관리는 별도 탭으로 구분하고, 비활성화와 삭제의 의미를 시각적으로 구분한다.
- 위험한 행동은 영향과 보존 여부를 확인할 수 있는 흐름을 제공한다.

### 8.8 마이페이지

- 변경 가능 필드와 읽기 전용 필드를 시각적으로 구분하고, 업무 상태는 명시적 선택형으로 제공한다.
- 프로필 사진과 같은 보호된 정보는 읽기 전용으로 처리한다.

## 9. 로딩·빈 상태·오류

- 초기 로딩에는 화면 구조를 예측할 수 있는 상태를 제공한다.
- 빈 상태는 오류처럼 표현하지 않고 가능한 다음 행동을 안내한다.
- 오류는 발생 위치 가까이에 표시하고 가능한 경우 재시도 수단을 제공한다.
- 인증 만료는 작업 내용 손실을 최소화하면서 재인증 흐름으로 안내한다.
- 권한 거부는 기능이 존재하지 않는 것처럼 위장하지 않고 접근할 수 없는 이유를 적절히 알린다.

## 10. 접근성 기준

- 모든 기능은 키보드로 접근 가능해야 한다.
- 포커스 표시를 제거하지 않는다.
- 시맨틱 Landmark와 Heading 순서를 유지한다.
- 입력과 오류 메시지를 프로그램적으로 연결한다.
- Modal과 Side Panel의 포커스를 관리한다.
- 색상 대비는 적용 가능한 WCAG AA 수준을 목표로 한다.
- 상태와 정보 전달을 색상에만 의존하지 않는다.
- 캘린더와 타임라인에는 목록 또는 텍스트 기반 대체 접근 경로를 고려한다.

## 11. 콘텐츠 규칙

- 버튼은 `확인`보다 `일정 저장`, `예약 취소`처럼 결과가 드러나는 동사를 사용한다.
- 날짜·시간과 상태 표현을 화면마다 일관되게 사용한다.
- 오류 메시지는 원인과 사용자가 취할 행동을 설명한다.
- 내부 코드, 예외, API 용어를 사용자 메시지에 그대로 노출하지 않는다.
- 파괴적 행동은 취소할 수 없는 결과를 명확히 설명한다.

## 12. 디자인 검증 체크리스트

- Desktop과 Mobile에서 핵심 흐름을 수행할 수 있는가
- Loading, Empty, Error, Permission 상태가 있는가
- 키보드만으로 조작할 수 있는가
- Modal과 Panel의 포커스가 올바르게 이동하는가
- 색상 없이도 상태를 이해할 수 있는가
- 변경 불가 필드가 명확한가
- 위험한 행동에 확인 절차가 있는가
- API 지연 또는 실패 중 중복 제출이 방지되는가

## 13. 미결정 사항

- 확정 디자인 토큰과 전체 색상 팔레트
- Typography와 아이콘 세트
- 정확한 반응형 Breakpoint
- 캘린더 및 타임라인 UI 라이브러리
- 헤더 검색 대상과 동작
- 회의실 검색 순위 표시 방식
- AI 응답 근거와 실행 확인 UI
