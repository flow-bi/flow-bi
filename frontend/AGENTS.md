# Frontend AGENTS.md

## 1. 목적과 적용 범위

이 문서는 `frontend/`에서 작업하는 사람과 Frontend Agent가 따라야 하는 영역별 규칙을 정의한다. 루트 `AGENTS.md`를 구체화하며, 상위 보안·아키텍처·Active Plan 규칙을 완화하지 않는다.

적용 대상:

- React 및 TypeScript 제품 코드
- 화면, 컴포넌트, 폼과 사용자 상호작용
- API 클라이언트와 서버 상태 연결
- 프런트엔드 테스트와 문서
- Tailwind CSS 기반 스타일

## 2. 작업 전 필수 확인

Frontend Agent는 변경 전에 다음을 MUST 확인한다.

1. 루트 `AGENTS.md`, `ARCHITECTURE.md`, `SECURITY.md`, `CONVENTIONS.md`
2. 승인된 Active Plan과 담당 Task
3. 관련 Product Spec과 Design Doc
4. `frontend/FRONTEND.md`와 `frontend/DESIGN.md`
5. 관련 API 계약인 `backend/API.md`
6. `docs/quality/quality-model.md`
7. 수정 가능·금지 경로, 완료 조건과 검증 항목

API 계약이 없거나 화면 요구사항과 충돌하면 응답 형식을 추측해 고정하지 않는다. Mock 또는 임시 타입을 사용할 경우 Active Plan과 작업 결과에 임시 상태를 기록한다.

## 3. 작업 범위

- Frontend는 `frontend/`만 수정한다.
- Backend, DB 스키마, Harness, 루트 정책 문서는 `frontend/` 범위를 벗어나므로 직접 수정하지 않고, 기록해두었다가 담당 Agent가 수정한다.
- API 변경이 필요하면 백엔드 코드를 임의로 수정하지 않고 계약 변경 필요사항을 보고한다.
- 관련 없는 화면 정리, 전역 스타일 변경, 의존성 교체를 함께 수행하지 않는다.
- 디자인 시안에 없는 신규 UX를 임의로 확정하지 않는다.

## 4. TDD 실행 규칙

프런트엔드 코드 구현과 버그 수정은 루트 TDD 규칙을 따른다.

- 사용자에게 보이는 동작과 접근 가능한 인터페이스를 실패 테스트로 먼저 표현한다.
- 컴포넌트 내부 구현보다 사용자의 입력, 동작과 출력 결과를 검증한다.
- 서버 통신은 통제 가능한 경계로 대체하되 실제 API 계약과 다른 Mock을 만들지 않는다.
- 로딩, 빈 결과, 성공, 검증 오류, 권한 거부, 서버 오류 상태를 적용 가능한 범위에서 검증한다.

## 5. 상태 관리

- 서버 데이터, 로딩, 오류, 캐시와 재조회는 TanStack Query가 관리한다.
- 여러 화면에서 공유되는 순수 클라이언트 상태만 Zustand에 둔다.
- 입력 중인 폼 값은 React Hook Form으로 관리한다.
- 단일 컴포넌트 또는 가까운 트리에서만 사용하는 UI 상태는 로컬 상태로 유지한다.
- 같은 서버 데이터를 Query Cache와 Zustand에 중복 저장하지 않는다.
- 서버 응답을 임의로 낙관적 갱신할 경우 실패 복구 전략과 테스트가 있어야 한다.

## 6. 폼과 검증

- 폼은 React Hook Form과 Zod를 사용한다.
- 클라이언트 검증은 사용자 경험을 위한 것이며 백엔드 검증을 대체하지 않는다.
- 필수값, 허용 길이, 날짜·시간 범위와 상호 의존 조건을 명시한다.
- 서버 오류를 필드 오류와 폼 전체 오류로 구분해 표시한다.
- 오류 메시지는 사용자가 수정 방법을 알 수 있도록 작성한다.
- 제출 중 중복 요청을 방지하고 결과가 확정되기 전에 성공으로 표시하지 않는다.

## 7. 컴포넌트와 접근성

- 컴포넌트는 하나의 주요 책임을 가진다.
- 공통화는 실제 반복과 안정된 인터페이스가 확인된 뒤 수행한다.
- 시맨틱 HTML을 우선하고 클릭 가능한 비시맨틱 요소로 버튼을 대체하지 않는다.
- 모든 폼 입력은 연결된 레이블을 가진다.
- 모달과 우측 패널은 키보드 포커스 이동, 닫기, 복귀를 지원해야 한다.
- 색상만으로 상태를 전달하지 않는다.
- 로딩과 오류 상태를 보조 기술이 인식할 수 있어야 한다.

## 8. 보안과 개인정보

- 프런트엔드에 비밀정보를 포함하지 않는다.
- 권한에 따른 메뉴 숨김은 UX 처리이며 실제 인가는 서버가 수행한다.
- 토큰 저장과 전달 방식은 승인된 인증 Design Doc을 따른다.
- 사용자 입력을 임의 HTML로 렌더링하지 않는다.
- 로그, 화면 오류, 개발 도구 메시지에 토큰이나 개인정보를 남기지 않는다.
- 다른 사용자의 일정·연락처 등은 서버가 허용한 범위만 표시한다.

## 9. API 연동

- API 접근 코드는 화면 컴포넌트에 흩어놓지 않고 기능별 API 경계에 둔다.
- 경로·요청·응답 타입은 `backend/API.md`와 일치해야 한다.
- HTTP 오류를 성공 데이터나 빈 배열로 위장하지 않는다.
- 인증 만료, 권한 거부, 검증 실패, 충돌, 서버 장애를 구분한다.
- 날짜·시간은 명시적인 형식과 시간대로 변환한다.
- 취소되거나 오래된 요청이 최신 화면 상태를 덮어쓰지 않도록 한다.

## 10. 완료 조건

Frontend Task는 다음 조건을 충족해야 한다.

- Active Plan과 Product Spec의 사용자 시나리오를 충족한다.
- TDD 증거가 있고, 적용 가능한 전체 테스트 통과한다.
- 타입 검사, 정적 분석, 빌드와 테스트를 적용 가능한 범위에서 통과한다.
- 주요 로딩·빈 상태·오류·권한 상태를 처리한다.
- 키보드와 기본 접근성 요구사항을 검증한다.
- API 계약과 다른 임시 구현이 남아 있으면 명확히 보고한다.
- 사용자 동작 또는 디자인 계약이 바뀌면 관련 문서를 갱신한다.

## 11. 금지 사항

- 서버 상태를 무분별하게 Zustand에 복제
- `any`로 타입 오류 은폐
- 권한 검사를 프런트엔드만으로 처리
- 테스트를 통과시키기 위한 단언 삭제 또는 약화
- API 계약 없이 응답 필드 임의 확정
- 전역 CSS로 특정 화면 문제를 우회
- 승인되지 않은 패키지 또는 UI 프레임워크 도입
- 민감정보의 로컬 저장·로그 출력

## 12. 작업 보고

Frontend Agent는 결과에 다음을 포함한다.

- 변경한 화면과 사용자 동작
- Red, Green, 최종 테스트 결과
- 처리한 로딩·오류·접근성 상태
- 사용하거나 가정한 API 계약
- 남은 UX, API 또는 디자인 문제
