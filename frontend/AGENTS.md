# AGENTS.md

## 문서 목적

이 문서는 `frontend/` 내부 작업에 적용되는 프론트엔드 전용 규칙이다.

루트 `AGENTS.md`의 공통 규칙을 먼저 따르고, 프론트엔드 작업에서는 이 문서의 규칙을 추가로 따른다.

---

## 프론트엔드 역할

프론트엔드는 다음을 담당한다.

- 화면 구현
- 라우팅
- 상태 관리
- API 연동
- UI 컴포넌트 구현
- mock 데이터 기반 화면 구성
- 사용자 입력 및 기본 검증 처리

백엔드 API, DB 구조, 인증/인가 로직 자체를 임의로 변경하지 않는다.

---

## 먼저 확인할 문서

프론트엔드 작업을 시작하기 전에 아래 문서를 확인한다.

1. `docs/PRODUCT.md`
2. `docs/FRONTEND.md`
3. `docs/DESIGN.md`
4. `docs/API.md`
5. `docs/STATE.md`
6. `progress.md`

## 주요 화면

프론트엔드의 주요 화면은 다음과 같다.

- 로그인
- 대시보드
- AI 채팅
- 캘린더
- 조직도
- 회의실
- 관리
- 마이페이지
- 테마 설정

## 작업 기본 원칙

- 한 번에 하나의 화면 또는 하나의 기능만 작업한다.
- 기존 폴더 구조와 컴포넌트 구조를 우선 따른다.
- 요구사항에 없는 화면이나 기능을 임의로 추가하지 않는다.
- 라우팅 구조는 `docs/FRONTEND.md`를 따른다.
- UI 스타일과 컴포넌트 기준은 `docs/DESIGN.md`를 따른다.
- API 요청/응답 기준은 `docs/API.md`를 따른다.
- 상태 관리는 `docs/STATE.md`를 따른다.
- API가 아직 없는 경우 mock 데이터를 사용하되, 실제 API로 교체 가능한 구조로 작성한다.
- 작업 후 변경 내용과 확인 결과를 `progress.md`에 기록한다.
- 주요 화면은 Loading, Empty, Error 상태를 고려한다.
- API 응답을 확인한 뒤 성공 또는 실패 UI를 표시한다.

## 캘린더 규칙

일정 유형은 다음 세 가지로 구분한다.

- 개인 일정
- 팀 일정
- 프로젝트 일정

일정 등록 폼에는 다음 항목을 포함한다.

- 제목
- 날짜
- 시작 시간
- 종료 시간
- 하루 종일 여부
- 위치
- 일정 유형
- 공개 범위
- 색상 라벨
- 참석자
- 설명

일정 유형별 조건은 다음과 같다.

- 개인 일정: 본인 기준 일정으로 등록한다.
- 팀 일정: 팀 선택이 필요하다.
- 프로젝트 일정: 프로젝트 선택이 필요하다.

---

## 회의실 규칙

회의실 화면은 다음 기준을 따른다.

- 회의실 목록과 예약 현황을 분리해서 표현한다.
- 날짜, 시간, 인원, 상태 기준으로 검색할 수 있어야 한다.
- 예약 버튼 클릭 시 우측 드로어 또는 모달로 예약 폼을 표시한다.
- 예약 시간 중복 여부는 백엔드 검증 결과를 우선한다.
- 프론트엔드는 필수값, 시간 범위, 인원 수 등 기본 입력값만 1차 검증한다.

---

## AI 채팅 규칙

AI 채팅 화면은 다음 기준을 따른다.

- 기본 단축어를 제공한다.
- 사용자가 직접 자연어 요청을 입력할 수 있다.
- AI 응답은 채팅 메시지 형태로 표시한다.
- 일정 등록, 회의실 예약 등 데이터 변경 작업은 확인 UI를 거친다.
- API가 완성되기 전까지는 mock 응답을 사용할 수 있다.
- mock 응답은 실제 AI 응답과 구분 가능하게 관리한다.

---

## mock 데이터 규칙

- mock 데이터는 실제 API 응답 구조와 최대한 유사하게 작성한다.
- mock 데이터는 실제 API 연동 시 쉽게 제거할 수 있어야 한다.
- mock 데이터 사용 여부는 `progress.md`에 기록한다.
- mock 데이터를 실제 서비스 데이터처럼 고정하지 않는다.
- UI 컴포넌트 내부에 큰 mock 데이터를 직접 두지 않는다.

---

## 페이지 리팩터링 가이드

### 요약

- 각 페이지의 `*.tsx` 파일은 최대한 간결하게 유지한다.
- 읽기 좋은 UI 블록은 페이지 내부 컴포넌트로 분리한다.
- 페이지에서만 사용하는 mock data, type, lib, constants는 해당 페이지 폴더 안으로 이동한다.
- 여러 페이지에서 공유하는 데이터, 유틸, 컴포넌트는 기존 전역 폴더에 유지한다.
- 사용자가 특정 페이지 리팩터링을 요청하면, 별도 지시가 없는 한 해당 페이지 하나만 리팩터링한다.

---

## 폴더 규칙

페이지 기본 구조는 아래 형태를 따른다.

```txt
src/pages/{page}/{PageName}.tsx
src/pages/{page}/components/*.tsx
src/pages/{page}/mock/*.ts
src/pages/{page}/types/*.ts
src/pages/{page}/lib/*.ts
src/pages/{page}/constants/*.ts
```

## Feature Plan Rules

기능 단위 작업을 시작할 때는 `docs/plans/_template.md`를 기준으로 작업 계획 파일을 작성한다.

작업 중인 계획 파일은 `docs/plans/active/`에 둔다.

예시:

- `docs/plans/active/dashboard.md`
- `docs/plans/active/rooms.md`
- `docs/plans/active/calendar.md`

계획 파일에는 다음 내용을 기록한다.

- 작업 목적
- 작업 범위
- 관련 문서
- 화면 요구사항
- 필요한 데이터
- UI 상태
- 구현 단계
- Codex 프롬프트 기록
- 작업 결과 기록
- 완료 기준
- 회고

작업이 완료되면 해당 계획 파일의 상태를 `Done`으로 변경하고, 필요하면 `docs/plans/completed/`로 이동한다.

`progress.md`에는 전체 진행 상황 요약만 기록하고, 기능별 상세 기록은 `docs/plans/active/` 또는 `docs/plans/completed/`에 남긴다.

## 문서 기록 규칙

- `frontend/docs/plans/active/*.md` 파일은 작업 계획 문서로 사용한다.
- active 계획 문서에는 구현 완료 기록이나 진행 로그를 추가하지 않는다.
- active 계획 문서에는 작업 범위, 요구사항, 작업 목록, 완료 기준, 계획 변경 사항만 작성한다.
- 실제 구현 내역, 수정 파일, 확인 결과, 남은 이슈는 `frontend/progress.md`에 기록한다.
- 기능 작업 완료 후에는 반드시 `frontend/progress.md`를 갱신한다.
- `dashboard.md`, `rooms.md`, `calendar.md`, `organization.md` 등 active 계획 문서에 progress 로그를 남기지 않는다.
