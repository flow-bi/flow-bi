# 프론트엔드 페이지 리팩터링 가이드

## 요약

- 각 페이지의 `*.tsx` 파일은 최대한 간결하게 유지하고, 읽기 좋은 UI 블록은 페이지 내부 컴포넌트로 분리한다.
- 페이지에서만 사용하는 mock data, type, lib, constants는 해당 페이지 폴더 안으로 이동한다.
- 여러 페이지에서 공유하는 데이터, 유틸, 컴포넌트는 기존 전역 폴더에 유지한다.
- 사용자가 특정 페이지 리팩터링을 요청하면, 별도 지시가 없는 한 해당 페이지 하나만 리팩터링한다.

## 폴더 규칙

- 페이지 기본 구조는 아래 형태를 따른다.
  - `src/pages/{page}/{PageName}.tsx`
  - `src/pages/{page}/components/*.tsx`
  - `src/pages/{page}/mock/*.ts`
  - `src/pages/{page}/types/*.ts`
  - `src/pages/{page}/lib/*.ts`
  - 페이지 전용 constants가 있으면 `src/pages/{page}/constants/*.ts`
- 기존 `*.mock.ts`, `*.types.ts`, `*.lib.ts`, `*.constants.ts` 파일이 페이지 전용이면 위 구조에 맞는 페이지 내부 폴더로 이동한다.
- 여러 페이지에서 사용하거나 앱 공통 관심사에 해당하는 파일은 `src/mocks`, `src/types`, `src/lib`, `src/constants`에 유지한다.
- 공용 컴포넌트는 `src/components/shared` 또는 기존 공용 컴포넌트 위치에 유지한다.

## 리팩터링 규칙

- `Page.tsx`는 상태 관리, 이벤트 핸들러, 주요 데이터 계산, 페이지 조립 역할에 집중한다.
- 카드, 리스트, 테이블, 탭, 사이드 패널, 모달, 폼 영역, 메시지/입력 영역처럼 반복되거나 큰 UI 블록은 `components/`로 분리한다.
- 분리한 컴포넌트에는 필요한 데이터와 핸들러를 명시적인 props로 전달한다.
- 페이지 전용 타입은 해당 페이지의 `types/` 폴더에서 관리한다.
- mock data는 UI 컴포넌트 안에 두지 않고 `mock/`으로 이동한다.
- 날짜 계산, 필터링, 정렬, 포맷팅 같은 순수 로직은 `lib/`로 이동한다.
- 사용자가 화면이나 동작 변경을 요청하지 않았다면 기존 UI 텍스트, 스타일, 동작을 유지한다.
- import는 기존 방식에 맞춰 `@/*` alias와 상대 경로 import를 일관되게 사용한다.

## 품질 기준

- 리팩터링 후 사용자가 보는 화면과 주요 상호작용은 기존과 동일해야 한다.
- 과도하게 쪼개지 않는다. 한 번만 쓰이는 블록이라도 페이지 가독성을 크게 개선하면 분리하되, 단순 태그 수준의 작은 요소까지 컴포넌트화하지 않는다.
- 페이지 전용 파일을 불필요하게 페이지 폴더 밖으로 노출하지 않는다.
- 새 컴포넌트 파일명은 역할이 드러나는 PascalCase로 작성한다.

## 기본 전제

- 사용자가 페이지 이름을 지정하면 해당 페이지만 리팩터링한다.
- 여러 페이지에서 사용하는 전역 파일은 이동하지 않는다.
- 페이지 전용으로 판단되는 파일만 페이지 내부 `mock`, `types`, `lib`, `constants` 폴더로 이동한다.
