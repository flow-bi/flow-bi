# Core Beliefs

이 문서는 프로젝트 전반에서 반복적으로 적용되는 설계 원칙만 기록한다. 기능 설계, API 설계, DB 설계, 화면 설계는 각 전용 문서에서 다룬다.

## 1. 문서는 판단의 기준이어야 한다

### 원칙

요구사항과 설계 판단은 문서에 기록된 내용을 기준으로 한다. 문서에 없는 내용은 확정된 사실처럼 다루지 않는다.

### 이유

이 프로젝트는 개발자와 AI가 같은 기능을 동일하게 이해하는 것을 목표로 한다. 기억, 추측, 구두 합의보다 문서화된 근거가 우선되어야 한다.

### 적용 예시

AI가 일정 생성 기능을 해석할 때 `calendar.md`와 관련 reference를 먼저 확인하고, 미확정 항목은 임의로 보완하지 않는다.

## 2. 기능 요구사항과 설계 원칙은 분리한다

### 원칙

개별 기능의 동작은 Product Spec에 둔다. Core Beliefs에는 여러 기능에 반복 적용되는 판단 기준만 둔다.

### 이유

같은 요구사항을 여러 문서에 반복하면 변경 시 불일치가 생긴다. 문서의 역할을 분리해야 새 Design Doc이 추가되어도 중복이 줄어든다.

### 적용 예시

회의실 예약의 입력 항목은 `meeting-room.md`에 두고, Core Beliefs에는 일정과 예약의 책임 경계를 유지해야 한다는 원칙만 둔다.

## 3. 책임 경계를 먼저 정한다

### 원칙

새 요구사항은 먼저 어느 기능 또는 도메인이 소유하는지 결정한다. 연동 기능은 소유 책임과 참조 책임을 분리해 기록한다.

### 이유

그룹웨어 기능은 일정, 회의실, AI, 알림처럼 서로 연결된다. 책임 경계를 정하지 않으면 같은 규칙이 여러 기능에 중복되거나 서로 다르게 해석될 수 있다.

### 적용 예시

회의실 예약으로 일정이 생성되더라도 예약 행위의 소유 문서는 `meeting-room.md`이고, 캘린더 표시 기준은 `calendar.md`와 연결한다.

## 4. 인증된 사용자 맥락을 기본 전제로 삼는다

### 원칙

서비스 기능은 로그인한 사용자를 기준으로 판단한다. 사용자 이름, 권한, 일정, 알림, 소속 정보는 인증된 사용자 맥락에서 해석한다.

### 이유

그룹웨어는 회사 내부 사용자를 대상으로 하며 대부분의 데이터가 사용자와 연결된다. 인증 맥락이 없으면 접근 가능 여부와 개인화 기준을 판단할 수 없다.

### 적용 예시

대시보드의 오늘 일정과 헤더의 본인 이름은 익명 상태가 아니라 로그인한 사용자 기준으로 표시된다.

## 5. 민감 정보와 권한 판단은 일반 정보와 분리한다

### 원칙

비밀번호, 토큰 같은 인증 정보는 임직원 기본 정보와 분리한다. 기능 접근 여부는 사용자 속성이 아니라 역할 기반 권한 구조를 기준으로 판단한다.

### 이유

임직원 정보는 여러 기능에서 조회되지만 인증 정보와 권한 판단은 별도의 통제가 필요하다. 두 책임을 분리해야 조회 편의성, 보안, 권한 확장성을 함께 유지할 수 있다.

### 적용 예시

조직도에서 직원 정보를 보여줄 때 인증 관련 정보는 노출하지 않고, 관리자 탭 접근은 역할과 권한 기준으로 판단한다.

## 6. 여러 대상을 갖는 관계는 명시적으로 표현한다

### 원칙

하나의 업무 데이터가 여러 사용자, 팀, 프로젝트, 역할과 연결될 수 있으면 그 관계를 명시적으로 관리한다.

### 이유

다대다 관계를 단일 값처럼 다루면 공유 범위, 접근 권한, 변경 영향이 불명확해진다. 관계를 명시해야 기능 간 데이터 해석이 일관된다.

### 적용 예시

팀 일정이나 프로젝트 일정은 특정 개인 일정과 동일하게 취급하지 않고 공유 대상 관계를 기준으로 해석한다.

## 7. 자주 보는 정보와 자세히 보는 정보는 구분한다

### 원칙

목록, 요약, 현황처럼 자주 조회되는 정보와 상세 설명처럼 필요할 때 보는 정보는 같은 무게로 다루지 않는다.

### 이유

그룹웨어는 캘린더, 대시보드, 조직도, 회의실 현황처럼 반복 조회 화면이 많다. 조회 목적에 맞게 정보의 무게를 구분해야 사용성과 성능 요구를 함께 고려할 수 있다.

### 적용 예시

캘린더 목록에서는 일정 제목과 시간 중심으로 보여주고, 긴 설명은 상세 조회에서 확인하는 기준을 따른다.

## 8. AI와 미확정 기능은 확정된 경계를 넘지 않는다

### 원칙

AI는 기존 업무 기능에 자연어로 접근하는 경로로 본다. 질문, 확장, 충돌로 남아 있는 내용은 확정 전까지 설계 원칙이나 구현 범위로 승격하지 않는다.

### 이유

AI가 별도의 업무 규칙을 만들거나 미확정 항목을 임의로 채우면 Product Spec과 다른 동작이 생긴다. 불확실성은 숨기지 않고 미확정 사항으로 남겨야 한다.

### 적용 예시

AI가 자연어로 일정을 생성하더라도 일정 기준은 `calendar.md`를 따르며, 검색창 용도처럼 확정되지 않은 항목은 공통 설계 기준으로 사용하지 않는다.
