# design-docs/core-beliefs.md

이 문서는 개별 기능 스펙보다 상위에 있는, 프로젝트 전반의 설계 판단 기준을 기록한다. 새 기능을 설계할 때 아래 원칙과 충돌한다면, 기능을 원칙에 맞추거나 원칙 자체를 갱신하는 논의를 먼저 한다.

---

## 1. 외부 연동이 아닌 완전 자체 구축

이 그룹웨어는 기존에 운영 중인 외부 시스템을 대체하거나 통합하는 것이 아니라 **처음부터 만드는 신규 서비스**다. 따라서:

- 외부 그룹웨어/HR 시스템과의 어댑터, 동기화 레이어는 필요하지 않다.
- `users`, `teams` 등은 이 서비스가 유일한 원천(source of truth)이며, 관리자 CRUD API가 아직 없는 MVP1에서는 **DB 시딩**으로 초기 데이터를 만든다.
- 이 원칙 덕분에 아키텍처가 단순해졌다: 외부 API 장애 대응, 데이터 동기화 충돌 같은 문제를 고려할 필요가 없다.

## 2. MVP1은 "화면은 있지만 안 되는 기능"과 "실제로 쓸 수 있는 기능"을 명확히 구분한다

기능 하나하나를 전부 구현하는 대신, 각 화면을 다음 두 상태 중 하나로 명시적으로 분류한다:

- **실기능**: 실제 API와 연결되어 데이터가 저장/조회된다.
- **디자인만**: UI는 존재하지만 동작하지 않는다 (예: 관리 탭, 대시보드의 AI 요약).

이 구분은 `docs/product-specs/`의 각 문서 상단에 명시하며, "일단 화면만 만들어두고 나중에 붙이면 되지"라는 암묵적 가정으로 방치하지 않는다. 어떤 화면이 "디자인만"인지는 매번 팀이 명시적으로 확정한 것만 그렇게 표시한다.

## 3. 관리자 기능(RBAC 포함)은 실제 필요가 확인되기 전까지 만들지 않는다

요구사항정의서의 FR-001~008(직원/팀 관리), ERD의 RBAC 도메인(`roles`, `permissions` 등)은 설계는 되어 있지만 MVP1에서 구현하지 않는다. 100명 규모 조직을 가정한 MVP 단계에서는 시딩 데이터로 충분하고, 실제 관리 UI가 필요해지는 시점(MVP2 이후)에 다시 설계한다. 이는 게으름이 아니라, **당장 쓰이지 않을 기능에 도메인 복잡도(권한 체계, 관리 화면, 감사 로그)를 미리 쏟지 않는다**는 판단이다.

## 4. 도메인 간 결합은 서비스 레이어로만 허용한다

`auth`, `user`, `schedule`, `meetingroom` 네 도메인은 서로의 데이터를 필요로 하지만(예: 회의실 예약이 일정을 자동 생성), Repository/Entity를 직접 참조하지 않고 **서비스 인터페이스를 통해서만** 상호작용한다(ARCHITECTURE.md 3.2 참고). 이는 한 도메인의 내부 구현을 바꿔도 다른 도메인이 깨지지 않게 하기 위함이며, 3인이 각자 도메인을 나눠 작업할 가능성(현재는 개인별 실험 단계)을 고려한 선택이다.

## 5. 일정은 개인/팀/프로젝트에 동시에, 유연하게 연결될 수 있다

하나의 일정이 특정 개인, 팀 전체(하위 조직 포함), 프로젝트 멤버 전체에 동시에 공유될 수 있어야 한다는 요구를 `schedule_targets`라는 다형적(polymorphic) 매핑 테이블로 해결했다. 이 구조와 그 이유는 별도로 `schedule-sharing-model.md`에서 자세히 다룬다.

## 6. 원본 문서(요구사항정의서, 화면설계서, ERD)는 서로 어긋날 수 있고, 그럴 때 임의로 한쪽을 정답으로 가정하지 않는다

이 프로젝트는 문서 3종(요구사항정의서, 화면/기능 설계서, ERD설계서)이 서로 다른 시점에 작성되어 값의 정의가 어긋나는 경우가 실제로 여러 번 발견됐다(예: 프로필 사진 변경 가능 여부, `visibility` 컬럼의 값 체계). 이런 불일치를 발견하면:

1. 임의로 한쪽을 정답으로 가정해 조용히 넘어가지 않는다.
2. `product-specs/` 문서에 불일치 사실과 판단 근거를 명시적으로 남긴다.
3. 팀 확인을 거쳐 확정되면 관련된 모든 문서(ERD, product-specs, 필요 시 원본 요구사항정의서에 대한 각주)를 함께 갱신한다.

## 7. 인프라(Redis, 캐시 등)는 필요성이 실측되기 전까지 들이지 않는다

Redis는 애초에 refresh token 저장 용도로 고려됐으나, PostgreSQL의 `user_tokens` 테이블만으로 다중 기기 로그인 관리가 충분히 가능하다는 것이 확인되어 MVP1에서는 도입하지 않는다. "나중에 필요할 것 같아서" 미리 인프라를 추가하지 않고, 실제 병목(예: access token 즉시 무효화 필요, 조회 성능 저하 실측)이 나타나면 그때 도입한다 (ARCHITECTURE.md 6번 참고).

## 8. AI 기능은 별도 MVP 단계로 명확히 분리되어 있다

AI 챗봇/오케스트레이션은 데스크톱 상시 최상단(always-on-top) 플로팅 위젯 형태로 나중에 붙는 것을 전제로 하며, MVP1의 어떤 도메인 설계에도 AI를 위한 결합을 미리 만들지 않는다. 대시보드의 AI 요약처럼 화면에 자리는 있지만 기능은 없는 부분은 "디자인만"으로 명확히 분리해, MVP2에서 AI 오케스트레이션 레이어가 붙을 때 기존 도메인 구조를 크게 바꾸지 않고 확장할 수 있게 한다.
