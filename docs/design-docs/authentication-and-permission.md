# 인증 및 권한 Design Doc

이 문서는 인증, 세션, 권한 판단처럼 여러 기능에서 공통으로 참조하는 기술 설계 결정을 관리한다. 로그인 화면의 기능 요구사항은 `docs/product-specs/auth.md`, 관리자 기능 요구사항은 `docs/product-specs/admin.md`를 기준으로 한다.

## 적용 범위

- 로그인 이후 서비스 접근 전제
- 사용자 기본 정보와 인증 정보의 책임 분리
- 로그인 세션 및 토큰 관리 방향
- 최초 로그인 및 비밀번호 변경 정책
- 관리자 기능 접근을 포함한 역할 기반 권한 판단
- 시스템 관리자와 인사팀 관리자의 권한 범위

## 설계 결정

### 1. 인증된 사용자 맥락을 서비스 접근의 기본 전제로 둔다

#### 결정

그룹웨어의 주요 기능은 로그인한 사용자를 기준으로 동작한다. 사용자별 일정, 헤더 표시 이름, 마이페이지, 관리자 접근, 알림 정책은 인증된 사용자 맥락을 기준으로 판단한다.

#### 근거

- `requirements.md` NFR-001은 로그인 사용자만 서비스 이용 가능해야 한다고 정의한다.
- `core-beliefs.md`는 서비스 기능을 로그인한 사용자 기준으로 판단한다고 정의한다.
- `database.md`의 일정 등록자, 사용자 역할, 토큰, 소속 팀 정보는 사용자와 연결되어 있다.

#### 영향

개별 기능은 익명 사용자 흐름을 기본 전제로 두지 않는다. 인증이 필요한 화면과 데이터 조회는 사용자 식별 정보를 기준으로 접근 범위를 판단한다.

### 2. 사용자 기본 정보와 인증 정보는 분리한다

#### 결정

임직원 기본 정보와 비밀번호 해시, 토큰 같은 인증 정보는 같은 책임으로 다루지 않는다. 사용자 기본 정보는 조직도, 마이페이지, 관리 기능에서 사용하고, 인증 정보는 로그인과 세션 유지에 한정해 다룬다.

#### 근거

- `database.md`는 `users`를 임직원 핵심 정보 관리 테이블로 정의한다.
- `database.md`는 `user_credentials`를 비밀번호 해시 값을 인사 정보와 격리하기 위한 테이블로 설명한다.
- `requirements.md` NFR-002는 비밀번호를 암호화하여 저장해야 한다고 정의한다.
- `core-beliefs.md`는 민감 정보와 권한 판단을 일반 정보와 분리한다고 정의한다.

#### 영향

조직도, 직원 목록, 마이페이지 같은 사용자 정보 조회 기능은 인증 민감 정보를 함께 노출하지 않는다. 인증 기능 변경 시에도 사용자 기본 정보 기능의 책임과 분리해 검토한다.

### 3. 로그인 계정은 사용자와 1:1로 연결한다

#### 결정

한 명의 임직원은 시스템 내에서 하나의 고유한 인증 정보 세트를 가진다.

#### 근거

- `database.md`는 `users`와 `user_credentials`의 관계를 1:1로 설명한다.
- `database.md`는 `user_credentials.user_id`에 Unique 제약조건을 두어 한 사용자에게 하나의 인증 정보 세트만 연결되도록 설명한다.

#### 영향

인증 정보는 사용자 단위로 식별된다. 하나의 임직원에게 여러 자체 로그인 계정을 부여하는 흐름은 현재 확정된 설계 범위에 포함하지 않는다.

### 4. 로그인 세션은 여러 기기 사용을 고려해 관리한다

#### 결정

한 사용자는 여러 기기나 브라우저에서 로그인할 수 있는 세션 기록을 가질 수 있다.

#### 근거

- `database.md`는 `users`와 `user_tokens`를 1:N 관계로 설명한다.
- `database.md`는 PC, 태블릿, 스마트폰 등 여러 환경에서 각각 독립된 Refresh Token이 발급될 수 있어야 한다고 설명한다.
- `architecture.md`는 Authentication 기술로 JWT를 선택한다.

#### 영향

로그아웃, 토큰 만료, 기기별 세션 관리는 사용자 단일 세션만을 전제로 설계하지 않는다. 단, 구체적인 기기 관리 화면이나 모든 기기 로그아웃 기능은 현재 Product Spec에 확정되어 있지 않다.

### 5. 권한은 사용자 직접 권한이 아니라 역할 기반으로 판단한다

#### 결정

기능 접근 여부는 사용자에게 직접 세부 권한을 부여하는 방식이 아니라 역할을 통해 판단한다. 역할은 여러 권한을 가질 수 있고, 사용자는 여러 역할을 가질 수 있다.

#### 근거

- `database.md`의 RBAC 도메인은 사용자에서 역할, 역할에서 세부 권한으로 이어지는 구조를 정의한다.
- `database.md`는 `user_roles`, `role_permissions`를 다대다 관계 해소를 위한 매핑 구조로 설명한다.
- `core-beliefs.md`는 기능 접근 여부를 역할 기반 권한 구조로 판단한다고 정의한다.

#### 영향

관리자 탭 접근이나 특정 기능 제어는 사용자 이름, 직급, 팀만으로 판단하지 않는다. 권한이 필요한 기능은 역할과 권한 구조를 기준으로 접근 가능 여부를 확인한다.

### 6. 인증 방식은 자체 로그인만 제공한다

#### 결정

현재 서비스는 회사에서 발급한 계정과 비밀번호를 사용하는 자체 로그인 방식만 제공한다.

Google, Kakao, Microsoft 등 외부 인증 제공자를 이용한 소셜 로그인이나 기업 SSO는 현재 범위에서 제외한다.

#### 근거

- 현재 요구사항은 회사에서 관리하는 임직원 계정을 기준으로 로그인 기능을 정의한다.
- 사용자 계정과 인증 정보가 `users`, `user_credentials` 구조로 관리된다.
- 외부 인증 제공자 연동에 필요한 계정 연결, 정보 동기화, 장애 대응 정책이 별도로 정의되어 있지 않다.

#### 영향

- 로그인은 회사에서 발급한 계정 식별자와 비밀번호를 기준으로 처리한다.
- 외부 인증 제공자 계정과 내부 사용자 계정을 연결하는 기능은 구현하지 않는다.
- 추후 SSO 또는 외부 인증을 도입하려면 별도 Product Spec과 Design Doc을 작성한다.

### 7. 최초 로그인 시 비밀번호 변경을 강제한다

#### 결정

직원 계정 생성 시 회사가 초기 비밀번호를 부여한다.

사용자가 초기 비밀번호로 최초 로그인에 성공하면 일반 서비스 화면으로 이동하지 않고 비밀번호 변경 화면으로 이동한다.

사용자는 새로운 비밀번호로 변경을 완료한 후 기존 로그인 세션을 종료하고 다시 로그인해야 한다.

#### 최초 로그인 흐름

1. 인사팀 관리자가 직원 계정을 생성한다.
2. 시스템이 회사 정책에 따른 초기 비밀번호를 설정한다.
3. 사용자가 회사에서 부여받은 계정과 초기 비밀번호로 로그인한다.
4. 시스템이 최초 로그인 또는 비밀번호 변경 필요 상태인지 확인한다.
5. 비밀번호 변경이 필요한 사용자는 비밀번호 변경 화면으로 이동한다.
6. 사용자가 새로운 비밀번호를 입력하고 변경을 완료한다.
7. 기존 로그인 세션과 토큰을 만료 처리한다.
8. 사용자는 변경한 비밀번호로 다시 로그인한다.
9. 재로그인 성공 후 일반 서비스 화면에 접근한다.

#### 비밀번호 처리 기준

- 초기 비밀번호와 변경된 비밀번호는 평문으로 저장하지 않는다.
- 비밀번호는 단방향 해시 처리하여 저장한다.
- 비밀번호 변경 완료 전에는 일반 기능에 접근할 수 없다.
- 비밀번호 변경 성공 후 기존 Access Token과 Refresh Token은 더 이상 사용할 수 없도록 처리한다.
- 회사가 부여한 초기 비밀번호는 최초 로그인 이후 계속 사용할 수 없다.

#### 근거

- `requirements.md` NFR-002는 비밀번호를 암호화하여 저장해야 한다고 정의한다.
- 회사에서 생성한 초기 비밀번호를 계속 사용하면 계정 탈취 위험이 증가할 수 있다.
- 최초 로그인 시 본인만 아는 비밀번호로 변경하도록 하여 초기 인증 정보의 노출 위험을 줄인다.

#### 영향

- 사용자 인증 정보에는 비밀번호 변경 필요 여부를 판단할 수 있는 상태가 필요하다.
- 로그인 성공 이후에도 비밀번호 변경 여부에 따라 접근 가능한 화면이 제한된다.
- 비밀번호 변경 완료 시 기존 인증 토큰을 무효화해야 한다.
- 비밀번호 변경 화면과 재로그인 흐름은 `auth.md` 및 `my-page.md`와 연결된다.

### 8. 관리자 역할은 시스템 관리자와 인사팀 관리자로 구분한다

#### 결정

관리자 역할은 시스템 관리자와 인사팀 관리자로 구분한다.

각 역할은 담당 업무에 필요한 최소 권한만 가진다. 시스템 운영 권한과 직원·조직 관리 권한을 하나의 관리자 역할에 모두 포함하지 않는다.

#### 시스템 관리자

시스템 관리자는 서비스 운영과 기술 설정을 담당한다.

주요 권한은 다음과 같다.

- 시스템 로그 조회
- 서비스 운영 상태 조회
- 서비스 기능 활성화 및 비활성화
- 시스템 전역 설정 조회 및 수정
- 장애 및 오류 현황 확인
- 사용자 계정의 활성 상태 조회
- 관리자 기능 접근 이력 확인

시스템 관리자는 직원의 인사 정보, 소속 팀, 직급과 같은 인사 데이터를 임의로 수정하지 않는다.

#### 인사팀 관리자

인사팀 관리자는 직원과 조직 정보를 관리한다.

주요 권한은 다음과 같다.

- 직원 등록 및 조회
- 직원 기본 정보 수정
- 직원 계정 활성화 및 비활성화
- 직원 초기 비밀번호 발급 또는 재설정 요청
- 팀 생성 및 조회
- 팀 정보 수정
- 직원의 소속 팀 변경
- 직원의 직급 변경
- 조직도 정보 관리

인사팀 관리자는 시스템 로그, 서비스 기능 상태, 시스템 전역 기술 설정을 임의로 변경하지 않는다.

#### 공통 권한 원칙

- 관리 기능은 역할에 연결된 권한을 기준으로 노출하고 실행한다.
- 프런트엔드에서 관리자 메뉴를 숨기는 것만으로 권한 검증을 대체하지 않는다.
- 백엔드는 모든 관리자 요청에서 역할과 세부 권한을 검증한다.
- 한 사용자가 여러 관리자 역할을 가질 수 있다.
- 역할을 여러 개 가진 사용자는 각 역할에 포함된 권한을 합산하여 사용할 수 있다.
- 개인정보와 시스템 운영 정보는 담당 역할에 필요한 범위에서만 조회할 수 있다.

#### 근거

- 시스템 운영과 인사 정보 관리는 책임과 접근 대상이 다르다.
- 하나의 관리자 역할에 모든 권한을 부여하면 불필요한 개인정보 접근과 시스템 변경 위험이 커진다.
- 최소 권한 원칙을 적용하여 역할별 책임과 권한 범위를 분리해야 한다.

#### 영향

- 관리자 화면은 시스템 관리 기능과 인사 관리 기능을 구분해야 한다.
- 관리자 API는 역할명만 확인하지 않고 필요한 세부 권한까지 검증해야 한다.
- `roles`, `permissions`, `role_permissions`, `user_roles`에 역할별 권한 구성이 반영되어야 한다.
- 관리자 역할 또는 권한 변경은 보안과 개인정보 처리 범위에 영향을 주므로 명시적인 검토가 필요하다.

## Product Spec 연결

- `docs/product-specs/auth.md`: 로그인, 로그아웃, 최초 로그인, 비밀번호 변경, 인증 사용자 접근 제한
- `docs/product-specs/admin.md`: 관리자용 탭, 직원 관리, 팀 관리, 역할별 관리자 기능
- `docs/product-specs/my-page.md`: 본인 정보 조회 및 수정, 일반 비밀번호 변경, 알림 설정
- `docs/product-specs/organization-chart.md`: 직원 기본 정보 조회
- `docs/product-specs/calendar.md`: 등록자 기준 일정 수정·삭제, 사용자별 일정 조회
- `docs/product-specs/global-layout.md`: 관리자 권한에 따른 회사 대표 색상 테마 관리

## 미확정 사항

- 기기별 세션 관리 화면 제공 여부는 확정되어 있지 않다.
- 모든 기기에서 일괄 로그아웃하는 기능의 제공 여부는 확정되어 있지 않다.
- 초기 비밀번호의 생성 규칙과 전달 방법은 확정되어 있지 않다.
- 비밀번호 길이, 문자 조합, 변경 주기 등 상세 보안 정책은 확정되어 있지 않다.
- 시스템 관리자가 변경할 수 있는 서비스 기능과 전역 설정의 구체적인 목록은 확정되어 있지 않다.
