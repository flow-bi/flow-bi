# springdoc-openapi-llms.txt

이 프로젝트는 Spring Boot 3.x 기준으로 springdoc-openapi **2.x**(예: 2.8.x)를 사용한다. springdoc-openapi 3.x는 Spring Boot 4.x 전용이므로, Spring Boot 3.x를 쓰는 한 2.x 라인을 유지한다. (참고: SpringFox는 더 이상 유지보수되지 않으며 Spring Boot 3.x와 호환되지 않으므로 사용하지 않는다.)

## 의존성 (Gradle 기준, Spring Web MVC)

```gradle
dependencies {
    implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:2.8.5' // 버전은 Spring Boot 버전과의 호환 매트릭스 확인 후 조정
}
```

추가 설정 없이 동작하며, 기본적으로 다음 경로에서 확인 가능하다:
- OpenAPI JSON: `/v3/api-docs`
- Swagger UI: `/swagger-ui/index.html` (또는 `/swagger-ui.html` 리다이렉트)

`application.yml`에서 경로 커스터마이징:
```yaml
springdoc:
  api-docs:
    path: /api-docs
  swagger-ui:
    path: /swagger-ui.html
```

## Spring Boot 3.2+ 파라미터 이름 인식 이슈 (중요, 실제로 자주 겪는 문제)

Spring Boot 3.2부터 파라미터 이름 추론 방식이 바뀌면서, 컴파일 시 `-parameters` 옵션이 없으면 컨트롤러 메서드 파라미터명이 springdoc 문서에 제대로 안 나올 수 있다. Maven 기준:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <parameters>true</parameters>
    </configuration>
</plugin>
```

Gradle이라면 `build.gradle`의 `tasks.withType(JavaCompile)`에 `options.compilerArgs << '-parameters'` 추가.

## 컨트롤러 어노테이션 사용법 (AGENTS.md 5.3과 연결)

```java
@RestController
@RequestMapping("/api/v1/schedules")
@Tag(name = "Schedule", description = "일정 관리 API")
public class ScheduleController {

    @Operation(summary = "일정 생성")
    @ApiResponse(responseCode = "200", description = "생성 성공")
    @PostMapping
    public ApiResponse<ScheduleResponse> create(@Valid @RequestBody ScheduleCreateRequest request) {
        // ...
    }
}
```

- `@Operation(summary = "...")`: Swagger UI에 표시될 요약 설명. 모든 엔드포인트에 필수로 단다(AGENTS.md 5.3 규칙).
- `@Schema(description = "...", example = "...")`: DTO 필드 설명 및 예시값.
- `@Hidden`: 특정 컨트롤러/메서드를 문서에서 숨김 (health check, 내부 콜백 등에 사용, 보안 목적으로는 `@Hidden`만 믿지 말고 반드시 Spring Security로도 접근을 막을 것).

## Bean Validation과의 연동

`@NotNull`, `@Min`, `@Max`, `@Size` 등 JSR-303 어노테이션은 springdoc이 자동으로 스캔해서 문서에 제약조건으로 반영한다. 별도 설정 불필요.

## JWT 인증 스킴 문서화 (이 프로젝트의 auth 방식과 연결)

Swagger UI에서 "Authorize" 버튼으로 JWT를 테스트하려면 `@Bean`으로 전역 등록:

```java
@Configuration
public class OpenApiConfig {

    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
            .components(new Components().addSecuritySchemes("bearer-jwt",
                new SecurityScheme()
                    .type(SecurityScheme.Type.HTTP)
                    .scheme("bearer")
                    .bearerFormat("JWT")))
            .addSecurityItem(new SecurityRequirement().addList("bearer-jwt"));
    }
}
```

컨트롤러/컨트롤러 클래스 단위가 아니라 이렇게 전역 `@Bean`으로 등록하는 것을 권장한다(컨트롤러마다 어노테이션 다는 것보다 유지보수가 쉽다).

## API 그룹 분리 (도메인별로 문서를 나누고 싶을 때, 선택사항)

```java
@Bean
public GroupedOpenApi scheduleApi() {
    return GroupedOpenApi.builder()
        .group("schedule")
        .pathsToMatch("/api/v1/schedules/**")
        .build();
}
```

이렇게 하면 `/v3/api-docs/schedule`처럼 도메인별 스펙을 따로 볼 수 있다. MVP1처럼 도메인이 4개뿐인 규모에서는 필수는 아니지만, 프론트엔드가 도메인별로 타입을 생성할 때 유용할 수 있다.

## 프론트엔드 타입 동기화

OpenAPI 스펙(`/v3/api-docs`)이 생성되면, 프론트에서는 `openapi-typescript` 같은 도구로 타입을 자동 생성해서 쓰는 것을 권장한다(수동 타입 작성 금지, AGENTS.md 5.3). 구체적인 생성 스크립트/CI 연동 방식은 `frontend-mvp1.md` 실행 계획에서 확정한다.
