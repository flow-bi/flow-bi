# AGENTS.md

## 1. 목적

이 문서는 `flow-bi` 저장소에서 작업하는 사람과 AI Agent가 따라야 하는 최상위 실행 규칙을 정의한다.

프로젝트의 목표는 약 100명 규모 기업을 위한 AI 기반 그룹웨어를 개발하는 것이다. 주요 기능은 조직·직원 관리, 인증과 권한, 일정 관리, 회의실 예약, 알림, AI 업무 지원이다.

이 저장소는 Harness Engineering 방식을 사용한다. Agent는 자유롭게 범위를 확장하는 구현자가 아니라, 승인된 문서와 Active Plan을 근거로 제한된 작업을 수행하고 검증 가능한 결과를 남기는 작업자다.

## 2. 규칙의 강도

문서에서 사용하는 규칙의 의미는 다음과 같다.

- **MUST**: 반드시 지켜야 하며, 위반 시 작업 완료로 판정할 수 없다.
- **SHOULD**: 특별한 사유가 없다면 지켜야 한다. 위반 시 근거를 기록해야 한다.
- **MAY**: 상황에 따라 선택할 수 있다.

## 3. 지침 우선순위

이 문서는 충돌 해결 순서 자체를 정의하는 메타 규칙이다. 지침이 충돌할 때는 다음 순서로 적용한다.

1. 시스템이 강제하는 Permission 및 안전 정책
2. `SECURITY.md`
3. 사람의 현재 요청과 명시적 승인
4. 승인된 `docs/plans/active/`의 Active Plan
5. `ARCHITECTURE.md` 및 승인된 ADR
6. 관련 Product Spec
7. 관련 Design Doc
8. `docs/quality/quality-model.md`
9. `CONVENTIONS.md` 및 Frontend/Backend 영역 문서
10. 외부 Notion·Excel·참고 문서
11. Worker Prompt

외부 문서는 참고자료이며 내부 승인 문서를 덮어쓸 수 없다. Worker Prompt는 역할과 실행 방법을 구체화할 수 있지만 Active Plan의 범위를 넓힐 수 없다. 하위 `AGENTS.md`는 루트 규칙을 구체화할 수 있지만 완화하거나 무효화할 수 없다.

상충하는 지침을 발견하면 Agent는 임의로 선택하지 않고 작업 보고서에 기록한다. 보안, 권한, 요구사항 또는 작업 결과에 중대한 영향을 주는 충돌은 사람의 확인을 받아야 한다.

## 4. 작업 전 필수 절차

모든 Agent는 코드나 문서를 변경하기 전에 다음을 수행해야 한다.

1. 루트 `AGENTS.md`, `ARCHITECTURE.md`, `SECURITY.md`, `CONVENTIONS.md`를 읽는다.
2. 작업 대상 디렉터리에 하위 `AGENTS.md`가 있으면 읽는다.
3. 관련 Product Spec, Design Doc, Active Plan, ADR을 확인한다.
4. 작업 목표, 허용 범위, 금지 범위, 완료 조건, 검증 방법을 확인한다.
5. 요구사항이나 범위가 모순되면 구현 전에 모순을 기록한다.

Active Plan이 필요한 실행인데 대응하는 파일이 없거나 유효하지 않으면 Agent는 구현을 시작해서는 안 된다.

## 5. 작업 범위

Agent는 다음 원칙을 MUST 준수한다.

- Active Plan에 명시된 Task와 파일 범위만 변경한다.
- 필요한 최소 변경만 수행한다.
- 관련 없는 코드 정리, 대규모 리팩터링, 기술 교체를 함께 수행하지 않는다.
- 다른 작업자가 만든 변경을 되돌리거나 덮어쓰지 않는다.
- 생성 파일, 마이그레이션, 공개 API, DB 스키마 변경은 Plan에 명시된 경우에만 수행한다.
- 확정되지 않은 기술을 임의로 도입하지 않는다.
- 테스트 통과를 위해 요구사항이나 검증 규칙을 약화하지 않는다.

범위 밖의 문제를 발견하면 수정하지 않고 기술 부채 또는 후속 작업으로 기록한다. 단, 즉각적인 보안 사고나 데이터 손실 위험은 작업을 중단하고 사람에게 알린다.

## 6. 문서 우선 개발

다음 변경은 구현 전에 관련 문서가 존재하거나 같은 Task에서 함께 갱신되어야 한다.

- 사용자에게 보이는 기능 또는 동작 변경
- API 요청·응답·오류 계약 변경
- DB 테이블, 컬럼, 관계, 제약조건 변경
- 인증·인가·개인정보 처리 변경
- 모듈 경계나 의존 방향 변경
- 품질 게이트 또는 완료 조건 변경

현재 ERD와 `backend/DB_SCHEMA.md`는 초기 기준선으로 취급한다. 전체 문서 검토가 완료되기 전에는 구조 개선을 목적으로 임의 변경하지 않는다. 발견한 문제는 검토 대기 항목으로 기록한다.

## 7. 구현 원칙

- 비즈니스 규칙은 UI나 컨트롤러에만 두지 않고 해당 도메인에서 검증한다.
- 프런트엔드는 서버 상태와 클라이언트 상태를 분리한다.
- 백엔드는 도메인 간 의존성을 명시적으로 유지하고 순환 의존을 만들지 않는다.
- 모든 외부 입력은 신뢰하지 않고 경계에서 검증한다.
- 권한 검증은 프런트엔드 표시 제어만으로 대체할 수 없다.
- 시간 정보는 시간대와 경계 조건을 명확히 처리한다.
- 회의실 예약 등 동시성에 민감한 규칙은 서버와 데이터 저장소 수준에서 일관성을 보장해야 한다.
- 로그에 비밀번호, 토큰, 개인정보 또는 비밀정보를 남기지 않는다.

## 8. 검증 및 완료 조건

### 8.1 TDD 개발 원칙

코드 구현 및 버그 수정 Task는 TDD를 기본 개발 방식으로 MUST 사용한다.

- Worker는 `Red → Green → Refactor` 순서를 따라야 한다.
- `Red → Green → Refactor` 결과를 Task 실행 기록에 남긴다.
- 제품 코드를 작성하기 전에 기대 동작을 표현하는 실패 테스트를 먼저 작성한다.
- Red 단계에서는 테스트가 의도한 이유로 실패하는지 확인한다.
- Green 단계에서는 테스트를 통과시키는 데 필요한 최소 구현을 작성한다.
- Green 단계 이후 전체 테스트가 통과하는 상태에서 구조를 개선한다.
- 버그 수정은 문제를 재현하는 실패 테스트부터 작성하고 회귀 테스트로 유지한다.
- 테스트를 삭제하거나 단언을 약화하거나 검증을 우회하여 통과시켜서는 안 된다.
- 기존 동작을 변경하면 관련 테스트와 문서를 같은 Task에서 갱신한다.
- 문서 작성, 단순 설정 변경, 자동 생성 코드 등 TDD가 부적절한 작업은 예외로 둘 수 있다.
- TDD 예외는 구현 전에 Active Plan에 사유, 영향과 대체 검증 방법을 기록해야 한다.
- 테스트 도구가 확정되기 전에는 새 도구를 임의 도입하지 않는다. Active Plan에 승인된 대체 검증을 기록한다.

승인된 예외 없이 TDD 증거가 없는 코드 구현 또는 버그 수정 Task는 완료 처리할 수 없다.

### 8.2 공통 완료 조건

Agent의 자체 선언만으로 작업은 완료되지 않는다. Task는 최소한 다음 조건을 만족해야 한다.

- Active Plan의 인수 조건을 모두 충족한다.
- 변경 범위 검사를 통과한다.
- 적용 가능한 빌드, 정적 분석, 테스트를 통과한다.
- 보안 관련 변경은 `SECURITY.md`의 기준을 충족한다.
- API, DB, 사용자 동작이 바뀌었다면 관련 문서를 갱신한다.
- 실패한 검증이나 실행하지 못한 검증을 숨기지 않고 기록한다.
- 남은 위험, 기술 부채, 사람의 확인이 필요한 사항을 보고한다.

테스트 도구가 아직 확정되지 않은 영역에서는 검증을 생략했다고 표현하지 않는다. 실행 가능한 수동 또는 구조 검증 결과와 자동화되지 않은 항목을 명확히 구분한다.

## 9. Harness 실행 규칙

### 9.1 `harness-plan`

`harness-plan`은 관련 정책·요구사항·설계·품질 문서를 읽고 실행 가능한 Active Plan을 작성한다. Plan은 검증을 통과한 뒤에만 `docs/plans/active/`에 확정한다.

Active Plan 파일명은 다음 형식을 MUST 따른다.

```text
{feature-name}_{NN}.md
```

- `{feature-name}`은 영문 `kebab-case`를 사용한다.
- `{NN}`은 `01`부터 시작하는 2자리 버전이다.
- 예: `schedule-management_01.md`, `room-reservation_02.md`

Plan에는 최소한 다음 정보가 있어야 한다.

### 기본 정보

- 사용자 요청
- 작업 목적
- 작업 유형
- 관련 문서: Product Spec, Design Doc, ADR, API, DB 및 품질 문서
- 수정 가능 경로
- 수정 금지 경로

### 실행 Task

각 Task에는 다음 항목이 있어야 한다.

- Task ID: `Task 01`부터 시작하는 2자리 번호
- 담당 Worker: `harness/config/workers.json`에 등록된 Worker
- 선행 Task: 없으면 `none`
- 작업 목적
- 구현 항목
- 검증 항목
- 완료 조건: 필수 조건과 최소 `quality_score` 포함
- 실패 조건: 보안 위반, 범위 위반, 필수 검증 실패 포함
- 제외 범위
- 작업 결과: Plan 생성 시 `none`
- 남은 문제: Plan 생성 시 `none`

`작업 결과`와 `남은 문제`는 실행 중 Harness만 갱신한다. `quality_score`의 범위, 계산식과 최소 통과 기준은 `docs/quality/quality-model.md`를 따른다.

### 9.2 `harness-execute`

`harness-execute`는 유효한 Active Plan이 있을 때만 실행한다. Orchestrator는 Task별로 필요한 Worker를 선택하고 최소 문맥과 제한된 파일 범위를 전달한다.

Worker는 자신에게 배정된 Task만 수행한다. 작업 결과는 diff, 이벤트, 검증 결과, 리뷰, 피드백, 기술 부채와 함께 `harness/runs/`에 기록한다.

Task 실행은 최초 실행을 포함해 Task당 최대 3회로 제한한다.

- 실패한 검증 결과와 피드백을 다음 시도에 전달한다.
- 동일한 원인으로 실패한 시도도 횟수에 포함한다.
- 보안 또는 Permission 위반은 재시도하지 않고 즉시 중단한다.
- 3회 안에 완료 조건을 충족하지 못하면 해당 Task와 Plan을 완료 처리하지 않는다.
- Harness는 실패 상태, 시도 내역, 실패 원인과 남은 문제를 기록하고 사람의 확인을 요청한다.
- 3회를 초과한 추가 실행은 사람의 명시적인 승인 후 새로운 실행으로 시작한다.

## 10. 금지 행위

Agent는 명시적인 승인 없이 다음을 수행해서는 안 된다.

- 운영 환경 배포 또는 운영 데이터 변경
- 비밀정보 생성·노출·커밋
- 인증이나 권한 검사를 우회하는 코드 추가
- 데이터 삭제 또는 복구가 어려운 명령 수행
- DB 스키마의 파괴적 변경
- 의존성 또는 핵심 기술 스택 교체
- 요구사항 ID나 검증 실패 기록 삭제
- 리뷰를 통과한 것처럼 결과 조작

## 11. 사람의 승인이 필요한 변경

다음 항목은 구현 전 또는 완료 처리 전에 사람의 승인을 받아야 한다.

- 요구사항의 의미나 사용자 범위 변경
- 인증 방식, 권한 모델, 토큰 정책 변경
- 개인정보 보관 또는 노출 범위 변경
- 공개 API의 호환성을 깨는 변경
- DB 데이터 손실 가능성이 있는 마이그레이션
- 새로운 외부 서비스나 유료 API 도입
- AI가 사용자를 대신해 변경 작업을 실행하는 정책
- 품질 게이트의 완화 또는 검증 생략

## 12. 작업 보고

최종 보고에는 다음을 간결하게 포함한다.

- 수행한 변경
- 충족한 요구사항과 인수 조건
- 실행한 검증 및 결과
- 실행하지 못한 검증과 이유
- 남은 위험 또는 기술 부채
- 사람의 후속 결정이 필요한 사항
