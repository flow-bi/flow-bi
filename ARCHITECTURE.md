# ARCHITECTURE.md

## 1. 문서 목적

이 문서는 `flow-bi`의 상위 시스템 구조, 도메인 경계, 의존 방향과 기술적 제약을 정의한다. 세부 구현은 `frontend/FRONTEND.md`, `backend/BACKEND.md`, `backend/API.md`, `backend/DB_SCHEMA.md` 및 기능별 Design Doc에서 다룬다.

## 2. 시스템 개요

`flow-bi`는 약 100명 규모 기업을 대상으로 하는 웹 기반 그룹웨어다.

핵심 기능은 다음과 같다.

- 사번과 비밀번호 기반 로그인
- 직원·팀·직급·역할·권한 관리
- 월간·주간·일간 일정 관리
- 개인·팀·프로젝트 일정 공유
- 회의실 조회·검색·예약·취소
- 사용자 설정과 업무 상태 관리
- 일정 및 예약 알림
- AI 기반 일정 조회·생성 및 사내 문서 검색 확장

## 3. 기술 기준선

### Frontend

- React
- TypeScript
- Zustand
- TanStack Query
- React Hook Form
- Zod
- Tailwind CSS
- npm

### Backend

- Java
- Spring Boot
- PostgreSQL
- Redis
- JWT 기반 인증
- 도메인 기반 모듈 구조

### 미확정 영역

다음 기술은 아직 확정하지 않는다.

- AI 모델 및 AI API
- 문서 저장·검색·검색증강생성 방식
- 프런트엔드 및 백엔드 테스트 프레임워크의 세부 구성
- CI/CD 및 배포 환경
- 클라우드와 관측성 플랫폼

Agent는 미확정 영역에 기술을 임의 도입해서는 안 된다. 도입이 필요하면 Design Doc 또는 ADR과 사람의 승인이 선행되어야 한다.

## 4. 논리 아키텍처

```text
[Web Client]
     |
     | HTTPS / JSON API
     v
[Spring Boot Application]
     |
     +-- Identity & Access
     +-- Organization
     +-- Schedule
     +-- Project
     +-- Room Reservation
     +-- Notification
     +-- AI Assistant (future)
     |
     +-- PostgreSQL
     +-- Redis
     +-- External AI / Document Search (future)
```

프런트엔드는 사용자 상호작용과 화면 상태를 담당한다. 백엔드는 인증·인가, 비즈니스 규칙, 트랜잭션과 데이터 정합성의 최종 책임을 가진다.

## 5. 도메인 경계

### 5.1 Identity & Access

책임:

- 사번·비밀번호 로그인
- 비밀번호 변경
- Access Token 및 Refresh Token 수명주기
- 사용자 역할과 세부 권한 검증
- 로그아웃 및 세션 무효화

조직의 직급과 시스템 권한은 서로 다른 개념으로 유지한다.

### 5.2 Organization

책임:

- 직원, 팀, 조직 계층, 직급 관리
- 직원의 재직 상태와 업무 상태
- 조직도 및 직원 검색

재직 상태와 현재 업무 상태를 하나의 상태값으로 혼합하지 않는다.

### 5.3 Schedule

책임:

- 일정 생성·조회·수정·삭제 또는 취소
- 월간·주간·일간 조회
- 개인·팀·프로젝트 일정 분류
- 참석자, 공유 대상, 공개 범위 관리
- 종일 일정과 시간 일정 처리

일정 유형은 `PERSONAL`, `TEAM`, `PROJECT` 중 정확히 하나다. 하나의 일정은 여러 참석자와 연결될 수 있으며, 유형에 따라 여러 팀 또는 여러 프로젝트와 연결될 수 있다.

- 개인 일정은 작성자와 참석자에게 공개한다.
- 팀 일정은 연결된 하나 이상의 팀 소속 사용자와 참석자에게 공개한다.
- 프로젝트 일정은 연결된 하나 이상의 프로젝트 참여자와 참석자에게 공개한다.
- 팀 일정에는 프로젝트 공유 대상을, 프로젝트 일정에는 팀 공유 대상을 지정하지 않는다.

직원과 팀은 물리 삭제하지 않고 비활성화한다. 일정과 예약은 취소 상태를 기본으로 사용하고 삭제 기능이 필요한 경우 Soft Delete로 처리한다. 일반 기능 흐름에서 물리 삭제하지 않는다.

### 5.4 Project

책임:

- 프로젝트 정보와 상태
- 프로젝트 참여자
- 프로젝트 일정의 대상 판별

프로젝트 관리 기능의 확장은 별도 Product Spec을 통해 수행한다.

### 5.5 Room Reservation

책임:

- 회의실과 수용 인원·위치·장비 정보
- 날짜·시간·수용 인원 기반 회의실 조회
- 예약 생성·수정·취소
- 일정과 예약의 정합성
- 중복 예약 방지

중복 예약 판정은 UI의 사전 확인만으로 끝내지 않고 백엔드 트랜잭션 범위에서 보장해야 한다.

### 5.6 Notification

책임:

- 사용자별 알림 사용 여부 및 사전 알림 시간
- 일정과 회의실 예약 알림
- 발송 예약, 발송 상태, 실패와 재시도

알림 채널과 전달 인프라는 미확정이다. 도메인은 특정 외부 공급자에 직접 결합하지 않는다.

### 5.7 AI Assistant

책임:

- 자연어 질의 해석
- 사용자 권한 범위 내 일정 조회
- 확인을 거친 일정 생성 및 업무 작업
- 사내 문서 검색과 답변 근거 제공

AI Assistant는 추후 기능이다. AI는 기존 도메인 규칙을 우회하지 않으며, 사용자와 동일한 인증·인가된 애플리케이션 서비스를 통해서만 업무 기능을 실행한다.

## 6. 의존성 원칙

- 프런트엔드는 백엔드의 저장 구조에 의존하지 않고 API 계약에만 의존한다.
- 백엔드는 도메인을 먼저 분리하고 각 도메인 내부를 Controller, Service, Repository, Entity, DTO 계층으로 구성한다.
- 계층 의존은 `Controller → Service → Repository` 방향을 따른다.
- 도메인 간 직접 Repository 접근을 피하고 소유 도메인의 Service를 통해 협력한다.
- Redis와 AI 등 외부 시스템 접근은 Service가 조정하고 Controller에 기술 세부사항을 노출하지 않는다.
- 순환 의존은 허용하지 않는다.
- 공통 모듈에는 범용 값 객체와 기술 공통 기능만 두며 도메인 규칙을 숨기지 않는다.

구현 패키지의 상세 구조는 `backend/BACKEND.md`에서 정의한다.

## 7. 데이터 및 트랜잭션 원칙

- PostgreSQL을 영속 데이터의 기준 저장소로 사용한다.
- Redis는 캐시, 제한된 수명의 상태 또는 분산 제어에 사용하며 영속 데이터의 유일한 기준 저장소로 사용하지 않는다.
- DB 스키마 기준선은 `backend/DB_SCHEMA.md`에서 관리한다.
- 스키마 변경은 마이그레이션과 문서 갱신을 함께 수행한다.
- 하나의 비즈니스 작업에서 강한 정합성이 필요한 변경은 하나의 트랜잭션 경계로 처리한다.
- 캐시는 원본 데이터가 아니며 무효화 또는 만료 정책을 명시한다.
- 시간 구간은 시작을 포함하고 종료를 제외하는 `[start, end)` 규칙을 기본으로 한다.

현재 ERD는 전체 문서 작성 후 검토할 초기 기준선이다. Agent는 승인 없이 ERD 개선을 구현에 선반영하지 않는다.

## 8. API 원칙

- 클라이언트와 서버는 HTTPS 기반 JSON API로 통신한다.
- API는 리소스와 유스케이스의 의미가 드러나도록 설계한다.
- 요청 입력은 서버 경계에서 다시 검증한다.
- 성공과 오류 응답의 공통 형식은 `backend/API.md`에서 정의한다.
- 인증이 필요한 API는 사용자 신원을 검증하고 각 작업에 필요한 권한을 검사한다.
- 목록 API는 데이터 증가를 고려해 페이지네이션 또는 명확한 조회 범위를 가진다.
- 호환성을 깨는 API 변경은 사람의 승인과 마이그레이션 계획이 필요하다.

## 9. 프런트엔드 상태 원칙 (todo: frontend/AGENTS.md에도 포함되어 있는데 필요한가?)

- 서버에서 가져오는 데이터와 캐시는 TanStack Query가 관리한다.
- 여러 화면에서 공유하는 순수 클라이언트 상태는 Zustand가 관리한다.
- 로컬 UI 상태는 가능한 컴포넌트 가까이에 둔다.
- 폼 상태와 검증은 React Hook Form과 Zod를 사용한다.
- 서버의 권한 판정을 프런트엔드 상태로 대체하지 않는다.

## 10. 보안 아키텍처

- 비밀번호는 복호화 가능한 형태로 저장하지 않는다.
- Access Token은 짧은 수명을 가지며 Refresh Token은 별도로 관리한다.
- 인증과 권한 검증은 백엔드에서 강제한다.
- 관리자 기능은 일반 사용자 기능과 권한 경계를 분리한다.
- 개인정보와 인증정보는 최소한으로 조회하고 노출한다.
- AI 기능에도 동일한 권한과 감사 원칙을 적용한다.

세부 기준은 `SECURITY.md`를 따른다.

## 11. 품질 속성

아키텍처는 다음 품질 속성을 우선한다.

1. 보안과 권한 정확성
2. 데이터 정합성
3. 유지보수성과 변경 용이성
4. 사용자 기능의 정확성
5. 성능과 가용성
6. 관측 가능성과 감사 가능성

정량적 완료 기준은 `docs/quality/quality-model.md`에서 정의한다.

승인된 Active Plan은 이 문서와 ADR의 경계 안에서 작업을 구체화해야 한다. Active Plan은 아키텍처 또는 ADR을 암묵적으로 변경할 수 없으며, 변경이 필요하면 별도의 Design Doc 또는 ADR과 사람의 승인이 선행되어야 한다.

## 12. 아키텍처 변경 관리

다음 변경은 Design Doc 또는 ADR이 필요하다.

- 핵심 기술 스택 도입 또는 교체
- 도메인 경계와 의존 방향 변경
- 인증·권한 모델 변경
- DB 모델의 주요 구조 변경
- 외부 AI·검색·알림 서비스 도입
- 배포 토폴로지 또는 데이터 보관 정책 변경
- 성능을 위해 정합성 모델을 변경하는 결정
