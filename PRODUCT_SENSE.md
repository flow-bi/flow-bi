# PRODUCT_SENSE.md

`design-docs/core-beliefs.md`가 "기술적으로 왜 이렇게 설계했는가"를 다룬다면, 이 문서는 "기능/우선순위를 어떤 기준으로 판단하는가"를 다룬다. AI가 스펙에 없는 작은 결정(문구, 우선순위, 엣지 케이스 처리)을 내려야 할 때 이 문서를 기준으로 삼는다.

---

## 1. 타겟 유저

가상의 A기업(약 100명 규모)의 **일반 사무직 임직원**. 화면/기능 설계서와 요구사항정의서가 상정하는 사용자는 다음과 같은 특징을 가진다:

- 하루에도 여러 번 캘린더와 회의실 현황을 확인한다.
- IT에 익숙하지만 전문가는 아니다(복잡한 설정 화면을 좋아하지 않는다).
- 조직 개편이 잦지 않은 안정적인 회사 구조를 가진다(팀 계층은 자주 안 바뀜, 그래서 클로저 테이블 같은 구조를 도입해도 쓰기 부담이 적음).
- 관리자(HR/총무 담당자)는 소수이며, 이 서비스의 주 사용자는 아니다 — 그래서 관리자 기능(FR-001~008)이 MVP1에서 뒤로 밀려도 대다수 사용자 경험에 지장이 없다.

## 2. 이 서비스가 푸는 문제 (우선순위 순)

`프로젝트 소개` 문서의 "기존 문제"를 실제 판단 기준으로 재정렬하면:

1. **개인/팀 일정이 분산되어 있다** → 캘린더 통합이 가장 핵심 가치다. MVP1에서 가장 공들여야 할 화면.
2. **회의실 예약 현황을 실시간으로 확인하기 어렵다** → 두 번째로 중요. 캘린더와 강하게 연동되어야 한다(FR-021).
3. **사내 문서를 찾기 위해 반복 문의해야 한다** → AI 챗봇(문서 검색)이 해결할 문제이나, MVP2 이후. MVP1에서는 조직도 정도로 "누구에게 물어봐야 하는지"만 빠르게 찾게 해준다.
4. **중요한 일정 누락** → 알림 서비스가 해결할 문제이나, 화면설계서에 알림 화면이 없어 MVP1 스코프가 불확실하다(`product-specs/index.md` 미해결 이슈 참고). 우선순위상 1, 2번보다 낮다.

이 순서는 스코프가 애매한 기능을 만났을 때 "무엇을 먼저 살릴지" 판단하는 기준이 된다. 예를 들어 시간이 부족해 하나를 포기해야 한다면, 알림보다 캘린더/회의실 완성도를 우선한다.

## 3. 제품 판단 기준 (스펙에 없는 결정을 내릴 때)

### 3.1 "화려함보다 정보 파악 속도"
사내 도구이므로, 사용자는 오래 머무르지 않고 빠르게 필요한 정보를 확인하고 나간다. 화면을 설계할 때 "이 화면에서 사용자가 3초 안에 원하는 정보를 찾을 수 있는가"를 기준으로 삼는다(NFR-003의 3초 기준과 같은 정신이 UX에도 적용된다).

### 3.2 "실기능/디자인만 구분을 흐리지 않는다"
`design-docs/core-beliefs.md` 2번과 연결. 사용자 입장에서 클릭했는데 아무 일도 안 일어나는 것은 신뢰를 깎는다. "디자인만"인 화면(관리 탭, AI 요약)은 시각적으로도 비활성 상태임을 분명히 표시해서(`DESIGN.md` 4번 "관리 탭 시각적 구분"), 사용자가 헷갈리지 않게 한다.

### 3.3 "관리자보다 일반 사용자 경험을 먼저 완성한다"
MVP1에서 관리자 CRUD를 시딩으로 대체한 이유이기도 하다. 100명 규모 조직에서 관리자는 소수이고, 초기 데이터는 한 번 세팅하면 자주 바뀌지 않는다. 반면 일반 사용자는 매일 캘린더/회의실을 쓴다. 리소스가 제한적일 때는 항상 "더 많은 사람이 매일 쓰는 기능"을 우선한다.

### 3.4 "원본 문서가 애매하면, 사용자가 실제로 무엇을 하려는지로 판단한다"
예: 캘린더의 "공개 범위(개인/팀/프로젝트)" UI를 `schedule_targets` 공유 대상으로 재해석한 결정(`schedule.md`)은, 사용자가 실제로 하려는 행동("이 일정을 우리 팀과 공유하고 싶다")을 기준으로 데이터 모델을 맞춘 것이다. ERD의 원래 컬럼 정의를 기계적으로 따르기보다, 사용자 의도를 우선했다.

## 4. Non-goals (이 서비스가 아닌 것)

- **메신저/채팅 도구가 아니다**: 조직도의 "사람 클릭 시 채팅 연결"은 MVP1 제외 확장 기능이며, 자체 메신저를 만드는 것이 목표가 아니다.
- **전사 HR 시스템이 아니다**: 인사 발령, 급여, 근태 관리 등은 다루지 않는다. `users` 테이블은 순수히 그룹웨어 사용을 위한 최소 정보만 가진다(ERD설계서 명시).
- **범용 프로젝트 관리 도구가 아니다**: `projects` 테이블은 일정 공유 대상을 위한 것이지, Jira 같은 태스크/이슈 관리를 대체하지 않는다.

## 5. 이 문서를 언제 다시 봐야 하는가

- 새 기능 요청이 들어왔는데 우선순위가 애매할 때 (2번 문제 우선순위 참고)
- "이 정도는 그냥 만들어도 되지 않을까" 싶은 스코프 확장 유혹이 들 때 (4번 Non-goals 참고)
- 여러 구현 방식 중 어느 게 사용자에게 더 나은지 판단이 안 설 때 (3번 판단 기준 참고)
